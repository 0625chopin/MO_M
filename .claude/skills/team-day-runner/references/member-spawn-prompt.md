# 팀원 소환 / 재검증 프롬프트 템플릿

팀장이 `Agent(model: "sonnet", ...)` 로 팀원을 소환할 때와, 검증 이후 `SendMessage`로
수정·재검증을 요청할 때 쓰는 문구다. `<...>` 자리를 회차 실제 값으로 채운다.

---

## A. 최초 소환 프롬프트

```
너는 mo_im 팀의 팀원 <NAME>(파일: docs/team/<NN.NAME>.md)이다. 이번은 <N>일차 작업이다.

[읽기 범위 — 반드시 지킬 것]
- docs/team/<NN.NAME>.md (너의 프로필)와 docs/ROADMAP/team/<NN.NAME>.md (너의 담당 Task)만 읽어라.
- 전체 로드맵 docs/ROADMAP/ROADMAP.md 는 꼭 필요할 때만 참조한다.
- 다른 팀원의 docs/team/*·docs/ROADMAP/team/* 문서는 읽지 마라.
- 프로젝트 공통 규약(CLAUDE.md / AGENTS.md)은 항상 따른다: Next.js 16 관련 코드 전에
  node_modules/next/dist/docs/ 의 해당 버전 문서를 먼저 읽고, Mock First 개발과
  /sample 4상태(기본·로딩·빈·오류) 배치, D-030 전환 경계를 지킨다.

[이번 회차 담당 Task]
- <Task 번호와 제목 목록, 각 Task의 산출물 요약>

[선행 조건]
- <이 팀원이 이어받는 선행 산출물이 있으면 그 경로/내용을 명시. 없으면 "없음">

[산출물]
- 코드/컴포넌트는 <경로>에 만든다. 새 컴포넌트는 src/app/sample/page.tsx 쇼케이스에 등록한다.
- 실데이터 전환을 염두에 두고 표현/컨테이너를 분리하고, 구독은 인터페이스로 감싼다(D-030).

[보고]
- 작업을 마치면 (1) 완료한 Task 번호, (2) 만든/수정한 파일 경로 목록,
  (3) 남은 리스크나 다음 회차로 넘길 사항을 요약해 반환하라. 이 반환값이 곧 팀장 보고 내용이다.
```

## B. 이슈 수정 요청 (SendMessage — 최초 소환한 같은 에이전트에)

```
[<N>일차 교차검증 이슈] 리뷰 짝 <REVIEWER>가 아래를 지적했다. 수정하고 결과를 보고하라.
- <이슈 1: 파일:라인 또는 컴포넌트 + 무엇이 규약/요구사항 어디에 어긋나는지>
- <이슈 2 ...>
수정 후 (1) 고친 파일, (2) 각 이슈를 어떻게 해소했는지 한 줄씩 반환하라.
```

---

## C. 재검증 요청 (SendMessage — 리뷰 짝 에이전트에)

```
[<N>일차 재검증] <NAME>이 아래 이슈를 수정했다. 다시 교차검증하고
이슈 해소 여부를 항목별로 pass/fail 로 답하라. 새 이슈가 보이면 함께 적어라.
- <해소했다고 보고된 이슈 목록>
```

---

## 채우는 값 안내
- `<NN.NAME>` = 파일명 그대로. 예: `01.CORE`, `02.DESIGN`, `03.CREW`, `04.BOARD`.
- `<REVIEWER>` = 해당 팀원 프로필의 `리뷰 짝` 행에 적힌 상대(활성 팀원 중 상대 팀 담당).
- Task 번호·산출물은 1단계에서 산정한 이번 회차 배치에서 가져온다.
