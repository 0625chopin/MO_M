  schedule-planner 팀 편성/일정 산정, 워크로그 요약.

# team-day-runner

팀장(이 스킬을 실행하는 당신)이 **"N일차 진행"** 한 마디로 한 회차 분량의 팀 작업을
병렬 서브에이전트로 굴리고, 교차검증·문서갱신·워크로그·git 커밋까지 끝내는 절차다.

이 프로젝트의 계획 문서에는 "일차(Day)" 개념이 없다(스케줄 단위는 주차 1~45). 그래서
**"일차 = 진행 회차"** 로 다룬다 — 주차와 무관하게, 선행조건이 충족된 다음 Task 배치를
회차마다 하나씩 진행한다. `N`은 몇 번째 회차인지를 가리키는 카운터일 뿐이다.

---

## 0. 역할과 불변 원칙

- **팀장만** "N일차 진행" 지시를 처리한다. 실제 구현은 팀원 서브에이전트가 하고, 팀장은
  계획·소환·검증·문서화·커밋을 맡는다. 팀장이 직접 코드를 쓰지 않는다.
- **팀원 수 = `docs/team/` 하위 파일 수.** 현재 4명이며 파일명이 곧 팀원이다:
  `01.CORE.md`·`02.DESIGN.md`·`03.CREW.md`·`04.BOARD.md`. 인원을 코드에 박지 말고
  실행 때마다 `docs/team/`을 나열해 정한다(파일이 늘면 팀원도 는다).
- **팀원 소환은 Sonnet 모델**로 한다: `Agent(..., model: "sonnet")`.
- **팀원의 읽기 범위는 자기 파일뿐이다.** 각 팀원은 `docs/team/NN.<NAME>.md`(프로필)와
  `docs/ROADMAP/team/NN.<NAME>.md`(담당 Task)만 읽는다. 전체 로드맵
  `docs/ROADMAP/ROADMAP.md`는 **꼭 필요할 때만** 참조하고, **다른 팀원 문서는 읽지 않는다.**
  (단, 프로젝트 코딩 규약인 `CLAUDE.md`/`AGENTS.md`는 모든 에이전트에 항상 적용된다 —
  Next.js 16 버전 문서 선독, Mock First, `/sample` 4상태 등.)
- 문서 SSOT를 지킨다: 프로필=`docs/team/`, Task·공수·주차·의존=`docs/ROADMAP/team/`,
  전체일정=`docs/SCHEDULE/SCHEDULE.md`. 데이터를 복제하지 말고 각 소스를 참조/갱신한다.

아래 1→2→3→4 순서대로 진행한다. **4단계(완료 처리)는 3단계에서 이슈가 완전히 0이 된 뒤에만** 한다.

---

## 1. 착수 — 회차 판정과 shrimp 계획 수립

1. **N 확정.** 사용자 지시에서 회차 번호를 읽는다("3일차 진행" → N=3).
2. **완료 집합 파악.** `docs/dailyWorkLog/`의 기존 `*_Day.md`를 훑어 지금까지 완료된 Task 번호를
   모은다(디렉터리가 없으면 공집합 = 아직 아무 회차도 안 함). `N-1_Day.md`가 없는데 N≥2이면
   회차를 건너뛴 것이므로 사용자에게 알리고 계속할지 확인한다.
3. **이번 회차 대상 배치 산정.** `docs/ROADMAP/team/*.md`의 각 Task에서 `의존:`·`선행 대기:`
   필드만 읽어(팀원 파일 본문은 최소한으로) 아래를 만족하는 **미완료 Task**를 고른다:
   - `의존:`의 모든 Task가 완료 집합에 있음, **그리고**
   - `선행 대기:`의 모든 크로스팀 Task가 완료 집합에 있음.
   이 조건을 만족하는 Task 묶음이 이번 회차 배치이고, 그 Task를 담당하는 팀원이 **이번 회차 활성 팀원**이다.
   (`선행 대기: 없음`·`의존: 없음`인 Task는 항상 착수 가능 → 보통 1일차의 시작점.)
4. **shrimp-task-manager로 계획 수립.** 배치가 정해지면 곧바로:
   - `mcp__shrimp-task-manager__plan_task` 로 이번 회차 목표를 기술하고,
   - `mcp__shrimp-task-manager__split_tasks` 로 **팀원별 하위 태스크**로 쪼개 등록한다
     (태스크 이름에 담당 팀원과 원 Task 번호를 넣어 추적 가능하게).
   - 진행 중 상태 갱신은 `update_task`, 목록 확인은 `list_tasks`, 완료 판정은 `verify_task`.

산정 결과(활성 팀원, 각자의 담당 Task, 배치 내 선후 관계)를 짧게 사용자에게 보고한 뒤 2단계로 넘어간다.

---

## 2. 의존성 순서 spawn (릴레이)

이번 회차 배치 **안에서도** 팀원 간 선후가 있을 수 있다(A의 산출을 B가 받아 씀). 그래서 무조건
동시에 띄우지 않고 위상 정렬해서 소환한다.

- **선행이 없는 활성 팀원**은 한 메시지에 여러 `Agent` 호출로 **동시에** 소환한다.
- **선행이 있는 팀원**은 그 선행 팀원의 작업이 **끝난 뒤에** 소환한다(사용자 요구: "선행작업 완료 후 spawn").
  선행 완료 통지를 받고 나서 다음 팀원을 띄운다.
- 각 소환 프롬프트는 `references/member-spawn-prompt.md` 템플릿을 채워 쓴다. 반드시 포함:
  읽기 범위 제약(자기 파일만), 이번 회차 담당 Task 번호와 요약, 산출물 경로/형식, 프로젝트 규약 준수.
- **소환한 팀원의 agentId를 보관한다.** 이후 수정·재검증·종료에서 같은 에이전트를 이어 써야 컨텍스트가
  유지된다. 수정 지시는 새로 spawn하지 말고 `SendMessage`로 기존 에이전트에 보낸다.

모든 활성 팀원이 1차 산출을 끝내면 3단계로 간다.

---

## 3. 교차검증 루프 (이슈가 0이 될 때까지 반복)

활성 팀원끼리 작업 내용을 **상호 공유**하고 **교차 점검**한다. 누가 누구를 검증하는지는
각 팀원 프로필(`docs/team/NN.*.md`)의 **`리뷰 짝`** 행을 그대로 쓴다 — 이미 상대 팀끼리
교차하도록 설계돼 있다(예: CORE의 리뷰 짝 = DESIGN·BOARD). 리뷰 짝이 이번 회차에 활성이 아니면
활성 팀원 중 상대 팀 담당자에게 검증을 맡긴다.

검증 항목에는 산출물 정확성뿐 아니라 **프로젝트 규약 위반**도 포함한다(체크리스트):
- `/sample`에 컴포넌트가 등록됐고 **기본·로딩·빈·오류 4상태**가 토글로 보이는가.
- D-030 전환 경계 4가지(표현/컨테이너 분리, 구독 인터페이스, 도메인 오류 상태, 인증은 레이아웃)를 지켰는가.
- 사용자 노출 문자열 하드코딩 금지, 링크는 리소스 ID 기준(R-016/FR-052) 등 해당 FR/결정 준수.

**이슈가 발견되면 이슈가 사라질 때까지** 다음 6단계를 반복한다:
1. 이슈 내용을 **해당 팀원에게 전달**한다(`SendMessage`로 그 팀원 에이전트에).
2. 팀원이 **수정 작업**을 한다.
3. 수정 완료 후 **연관 팀원(리뷰 짝)이 교차 재검증**한다.
4. 이슈가 해결되면 **팀장에게 보고**한다.
5. 팀장이 **전체 테스트**를 돌린다: `npm run lint`, `npx tsc --noEmit`, `npm run build`.
   회차 내용과 무관하게 **세 명령을 항상 실행**하고 각 결과를 기록한다(문서만 바뀐 회차라도 실행해
   기존 코드가 깨지지 않았음을 보장한다). 실행 자체가 불가능한 상황(예: 의존성 미설치)이면 그 원인을
   먼저 해소한 뒤 실행한다.
6. 테스트에서 이슈가 나오면 1번으로 회귀, 없으면 검증 종료.

이슈가 0이고 전체 테스트가 통과해야 4단계로 넘어간다.

---

## 4. 완료 처리 (이슈 0 · 테스트 통과 시에만)

순서대로 수행한다.

1. **스케줄 문서 갱신.** 이번 회차에 완료한 각 Task에 대해 `docs/ROADMAP/team/NN.*.md`의
   해당 Task 불릿에 상태 마커를 추가/갱신한다:
   `- 상태: 완료 (N일차, YYYY-MM-DD)`
   (Task 상태의 SSOT는 이 문서다. 날짜는 시스템 오늘 날짜.)
2. **프로필 문서 갱신은 필요할 때만.** `docs/team/NN.*.md`는 팀원 상태(예: `상태` 행)가 실제로
   바뀔 때만 손댄다. 기본은 건드리지 않는다.
3. **워크로그 작성.** `docs/dailyWorkLog/N_Day.md`를 `references/worklog-template.md` 서식으로 만든다
   (디렉터리가 없으면 생성). 회차 요약·팀원별 완료 Task·교차검증 결과·발견/해결 이슈·테스트 결과·
   다음 회차에 열리는 Task를 담는다. **이 파일이 다음 회차의 완료 집합 판정 근거**가 되므로
   완료 Task 번호를 빠짐없이 명시한다.
4. **팀원 종료.** 보관해 둔 모든 활성 팀원 agentId에 `TaskStop`을 보낸다.
5. **git 커밋/푸시 (day-N 브랜치).**
   - 현재 브랜치에서 `git checkout -b day-N` (예: N=3 → `day-3`). main은 건드리지 않는다.
   - 변경을 스테이징하고 커밋한다. 커밋 메시지 말미에 다음 트레일러를 붙인다:
     ```
     Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
     Claude-Session: https://claude.ai/code/session_016NUwbhPTVeHrJ5NPa3NaV3
     ```
   - **push 전에 사용자 확인을 받는다.** 커밋까지 마친 뒤, 푸시할 브랜치(`day-N`)와 커밋 요약
     (변경 파일 수·완료 Task)을 사용자에게 보여주고 **"이 브랜치를 원격에 푸시할까요?"** 라고 명시적으로 묻는다.
     사용자가 승인하면 `git push -u origin day-N` 을 실행하고, 보류/거절하면 **커밋은 남겨 둔 채 푸시하지 않고**
     로컬 브랜치 상태만 보고한다.
   - 브랜치명 충돌 등으로 push가 막히면 사용자에게 상황을 보고하고 지시를 기다린다.

완료 후 사용자에게 **회차 요약**(활성 팀원, 완료 Task, 검증 결과, 만든 워크로그 경로, 푸시한 브랜치)을 보고한다.
