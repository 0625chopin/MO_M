# 문자열 모듈 — 추출 규약

**대상 Task**: ROADMAP Task 003. **근거**: NFR-023(M등급), D-011, R-016.
**단일 소스**: 이 문서와 `src/lib/strings/ko.ts`. 다른 문서에 이 규약을 복제하지 않는다.

## 1. 왜 필요한가

D-011로 v0.1은 한국어 단독이고 경로에 로케일 세그먼트를 두지 않는다(`/crews`, `/sample`).
번역 자체는 미루지만, 문자열을 컴포넌트에 하드코딩한 채로 화면을 늘리면 나중에 번역을 넣을 때
**전 컴포넌트를 다시 고쳐야 한다** — 이것이 R-016(로케일 경로 없이 시작해 다국어 도입 시 전 경로가
바뀌는 문제)이 지목한 위험 중 문자열 쪽 절반이다(나머지 절반은 경로 문제이며 §4에서 다룬다).
그래서 NFR-023은 "번역은 미루되 **추출은 v0.1부터**"를 요구하고, 이 모듈이 그 요구를 만족시킨다.

**지금 하는 일**: 문자열을 컴포넌트 밖으로 꺼내 키 구조로 정리한다.
**나중에 할 일**: 이 키 구조를 그대로 두고 로케일별 값 파일을 추가한다. 컴포넌트 코드는 건드리지 않는다.

## 2. 무엇이 대상인가

**대상**: 사용자가 화면에서 보는 모든 한국어 문자열.

- 버튼·링크 라벨, 페이지·섹션 제목
- 안내 문구, 빈 상태·오류 상태 메시지, 유효성 검증 실패 메시지
- 상태 배지(투표 진행 중/가결/부결/무효 등), 뱃지·칩 텍스트
- 토스트·알림 문구
- 폼 필드 라벨·placeholder
- `aria-label`, `alt` 텍스트 등 접근성 문자열(화면에 보이진 않아도 사용자에게 전달되는 문자열이므로 포함)
- 이메일/외부 발송 문구가 생기면 그것도 포함(현재는 없음)

**대상이 아닌 것**:

- 개발자용 콘솔 로그·에러 스택(`console.error("...")` 내부 등)
- 코드 식별자: CSS 클래스명, DB 컬럼명, API 필드명, 상태 머신의 상태값(`"open"`, `"closed_passed"` 등 —
  이건 로직 상수이지 사용자 문구가 아니다. 화면에 보여줄 한국어 라벨은 `vote.status.*`에 별도로 둔다)
- 코드 주석, 커밋 메시지, 문서 자체의 한국어(이 파일 포함)
- 순수 상수(단위 없는 숫자, ISO 코드 등)

판단이 애매하면 "이 문자열이 바뀌면 화면에 보이는 텍스트가 바뀌는가?"로 묻는다. 그렇다면 대상이다.

## 3. 디렉터리 구조

```
src/lib/strings/
  ko.ts       # 한국어 값 — 현재 유일한 로케일. 키 구조의 단일 소스.
  index.ts    # 접근 헬퍼(strings, t, getStrings)와 타입(Strings, Locale) 공개 API
  README.md   # 이 문서
```

- `src/lib/`에 두되 `utils.ts`처럼 범용적인 이름은 쓰지 않는다(shadcn CREW가 `utils.ts`를 따로 만든다).
  이 디렉터리 안에서만 그 이름을 써도 충돌하지 않는다.
- CORE가 `src/` 하위 디렉터리 규약을 이번 회차에 확정하기 전까지는 표준 Next.js `src/` 관례
  (`@/*` → `./src/*`)를 따라 `@/lib/strings`로 접근한다. CORE 규약이 이 위치와 다르게 확정되면
  이동만 하면 되고 내용은 바뀌지 않는다.
- 로케일이 늘면 `en.ts`처럼 같은 디렉터리에 나란히 둔다. `ko/`, `en/` 처럼 폴더로 쪼개는 것은
  파일이 도메인별로 분리될 만큼 커졌을 때(예: 컴포넌트 수 50개 이상) 재검토한다 — 지금은
  과설계다.

## 4. 키 명명 규칙

`ko.ts`는 **도메인(화면·기능) → 하위 그룹 → 리프** 3단 구조를 기본으로 한다.

```ts
export const ko = {
  board: {               // 최상위: 화면/기능 도메인. FR 절(4.D 커뮤니티 게시판 등)과 대략 맞춘다
    list: {               // 하위 그룹: 그 도메인 안의 화면/서브컴포넌트
      title: "게시판",     // 리프: 실제 문구
      empty: "아직 등록된 글이 없어요",
    },
  },
} as const;
```

- 키는 **camelCase**. 값(한국어 문자열)과 키(영문 식별자)를 분리해 번역 시 키는 그대로 두고
  값만 바꿀 수 있게 한다.
- 최상위 도메인은 요구사항 문서 4절의 기능 영역과 느슨하게 맞춘다: `common`, `nav`, `landing`, `home`,
  `crew`, `calendar`, `account`, `invitation`, `board`, `vote`, `meetup`, `chat`, `notification`,
  `auth`, `error`. 새 도메인이 필요하면 이 목록에 추가하고, 기존 도메인에 억지로 끼워 넣지 않는다.
- **상태 배지류는 상태 머신의 상태값과 키를 맞추되, 값은 별개다.** 예: `vote.status.closedPassed`의
  값은 `"가결"`이지만 이 값은 표시 문구일 뿐 판정 로직의 상태 문자열(`Task 009A`가 만드는 순수
  함수의 리턴값)과 같은 것이 아니다. 로직에서 이 문구를 직접 조합하지 말고, 로직이 반환한 상태값을
  키로 써서 문자열 모듈을 조회한다.
- 파라미터가 있는 문구는 `{paramName}` 자리표시자를 쓴다: `"참여 {voted}명 / 대상 {total}명"`.
  파라미터 이름은 호출부에서 넘기는 객체 키와 정확히 일치해야 한다(불일치 시 자리표시자가 그대로
  남는다 — 동작 근거는 `src/lib/strings/index.ts`의 `interpolate()` 주석 참고. §5도 `t()` 사용법을 다룬다).
- 하나의 문구가 여러 화면에서 완전히 동일하게 쓰이면 `common`에 두고 공유한다. **판단 기준은
  "같은 개체(entity)·같은 개념을 가리키는가"이지 "우연히 한국어 텍스트가 같은가"가 아니다.**
  예: "삭제된 게시글입니다"는 게시판 상세(`board.detail.deleted`)와 채팅 카드(`chat.postCard.deletedPost`)
  양쪽에서 **같은 게시글 엔티티**의 삭제 상태를 가리키므로 `common.post.deleted`에 한 번만 선언하고
  두 위치가 그 상수를 참조한다(`ko.ts`의 `common.post` 참고). 반대로 우연히 같은 한국어라도
  도메인 맥락이 다르면(예: 게시판의 "삭제"와 채팅 메시지의 "삭제" — 하나는 게시글, 하나는
  메시지) 각 도메인에 중복해서 둔다 — 나중에 하나만 번역이 달라질 수 있기 때문이다
  ("Delete post" vs "Delete message").
- **같은 상태값에 두 벌의 문구 세트를 만들지 않는다.** 상태 배지처럼 여러 화면(목록·상세)에서
  같은 상태 머신 값을 표시할 때도 문구 세트는 하나만 두고(예: `vote.status`), 화면마다 그중
  일부만 골라 쓰는 것은 괜찮지만 값 자체를 다시 정의하지 않는다. 축약 배지가 필요하면(예:
  목록은 6개 상태 중 4개만 노출) 그 이유를 주석으로 남긴다 — `ko.ts`의 `board.postType` 옆
  주석이 실제 사례다.

## 5. 접근 방법

```ts
import { strings, t } from "@/lib/strings";

// 파라미터 없는 문구 — 중첩 객체 직접 접근. 자동완성과 리네임이 그대로 동작한다.
strings.board.list.title;

// 파라미터가 있는 문구 — t()에 셀렉터 함수와 파라미터 객체를 넘긴다.
t((s) => s.vote.summary.quorum, { quorum: 4 });
```

`t()`가 `t("vote.summary.quorum")`처럼 문자열 경로를 받지 않고 셀렉터 함수 `(s) => s.vote.summary.quorum`를
받는 이유: 문자열 경로는 오타가 나거나 `ko.ts`에서 키 이름을 바꿔도 컴파일 타임에 잡히지 않는다.
셀렉터 함수는 `Strings` 타입(`typeof ko`)을 그대로 따라가므로 오타·키 이름 변경 누락이 즉시
타입 에러로 드러난다. 이는 "타입 안전한 접근 헬퍼"라는 산출물 요구를 문자열 경로 대신 만족시키는 방식이다.

컨테이너/서버 컴포넌트, 클라이언트 컴포넌트, Server Action 어디서든 같은 방식으로 쓴다 —
이 모듈은 순수 데이터 모듈이라 `'use client'` 경계와 무관하다.

## 6. 하드코딩 금지 규칙

**컴포넌트(`.tsx`) 안에 한국어 문자열 리터럴을 직접 쓰지 않는다.** JSX 텍스트 노드, `placeholder`,
`aria-label`, `title` 속성, 조건부로 조합하는 안내 문구 전부 대상이다.

- 이 규칙은 **아직 ESLint로 강제되지 않는다.** `eslint.config.mjs`는 CORE 담당이며 이 Task
  범위 밖이다. 지금은 **코드 리뷰에서 확인한다** — 리뷰 짝은 §8 체크리스트를 쓴다. 린트 규칙
  도입(예: `no-restricted-syntax`로 JSX 안 한글 리터럴 탐지)은 CORE가 판단할 별도 결정 사항이다.
- 예외적으로 허용되는 리터럴: 순수 상수(단위 없는 숫자), 개발자 전용 콘솔 로그, 아직 문자열
  모듈에 대응 도메인이 없어서 임시로 쓰는 경우는 예외가 아니라 **먼저 `ko.ts`에 도메인을 추가**한다.
- Mock 데이터(예: 더미 게시글 제목 "동네 러닝 모임 어때요?")는 문자열 모듈 대상이 **아니다** —
  그건 사용자 데이터이지 UI 문구가 아니다. UI 문구(빈 상태 안내, 버튼 라벨 등)와 혼동하지 않는다.

## 7. 새 컴포넌트를 만들 때 절차

1. `ko.ts`에 해당 도메인 네임스페이스가 있는지 확인한다. 없으면 §4 규칙대로 추가한다.
2. 필요한 문구를 리프 키로 추가한다(한국어 값 채움).
3. 컴포넌트에서 `strings.도메인.그룹.키` 또는 `t((s) => s.도메인.그룹.키, params)`로 접근한다.
4. 커밋 전 자체 점검: 컴포넌트 파일에 따옴표로 감싼 한국어 리터럴이 남아 있지 않은지 확인한다.

## 8. 리뷰 체크리스트 (리뷰 짝용)

- [ ] 새/수정된 `.tsx`에 하드코딩된 한국어 문자열이 없다 (JSX 텍스트, `placeholder`, `aria-label` 포함)
- [ ] 새 키가 §4 명명 규칙(도메인 → 그룹 → 리프, camelCase)을 따른다
- [ ] 상태 배지 문구의 키가 해당 상태 머신 값과 짝이 맞고, 같은 상태값에 두 벌의 문구 세트가
      생기지 않았다(§4) — 목록·상세처럼 화면이 여러 개면 하나의 문구 세트를 공유하고, 일부만
      쓰는 경우엔 그 이유를 주석으로 남겼는지 확인한다
- [ ] `{paramName}` 자리표시자를 쓰는 문구는 호출부의 파라미터 객체 키와 정확히 일치한다
- [ ] 같은 개체(entity)를 가리키는 완전히 동일한 문구를 두 도메인에 따로 선언하지 않고
      `common`에서 공유했다(§4). 우연히 같은 한국어라도 도메인 맥락(가리키는 개체)이 다르면
      공유하지 않고 각자 도메인에 뒀다(§4)
- [ ] Mock 더미 데이터(사용자 콘텐츠)를 문자열 모듈에 넣지 않았다(§6)

## 9. 게시글 링크는 이 모듈의 대상이 아니다 (FR-052 경계)

FR-052(게시글 링크 공유)는 채팅에 붙인 게시글 링크가 카드로 렌더될 때 **경로 문자열이 아니라
리소스 ID(`postId`) 기준으로 저장**해야 한다고 요구한다(R-016 완화책 ③). 이는 라우팅/링크
조립 로직의 책임이며, 이 문자열 모듈의 책임과 명확히 구분한다:

- **이 모듈이 다루는 것**: 카드에 표시되는 텍스트 — 제목 접두사, "삭제된 게시글입니다"
  (`chat.postCard.deletedPost`), "다른 크루의 게시글이에요"(`chat.postCard.otherCrewPost`) 등.
- **이 모듈이 다루지 않는 것**: 링크가 가리키는 실제 경로(`/crews/[id]/board/[postId]` 같은 URL
  조립). 경로 규칙이 바뀌어도(다국어 도입으로 `/ko/crews/...`가 되어도) 저장된 데이터가
  `postId`면 새 경로 규칙으로 다시 해석할 수 있다 — `postId`를 저장해 두면 경로가 바뀌어도
  깨지지 않는다는 것이 R-016 대응의 핵심이다. 경로 조립 헬퍼(예: `resolvePostPath(postId)`)는
  라우팅을 담당하는 팀/Task가 만들 별도 모듈이며 이 문자열 모듈에 두지 않는다.

## 10. 다국어 전환 시 실제로 할 일 (참고용 — 지금 하지 않는다)

1. `src/lib/strings/en.ts`를 만들고 `Strings`(=`typeof ko`) 타입을 만족하는 객체를 채운다.
   키를 하나라도 빠뜨리면 타입 에러로 드러난다.
2. `index.ts`의 `Locale` 유니온에 `"en"`을 추가하고 `dictionaries`에 `en: en`을 등록한다.
3. 로케일을 결정하는 지점(요청 헤더, 사용자 설정 등 — 아직 미정, D-011·NFR-024는 이번 범위
   밖이라 여기서 설계하지 않는다)에서 `getStrings(locale)`을 호출하도록 연결한다.
4. 컴포넌트 코드(`strings.x.y`, `t(...)` 호출부)는 **수정하지 않는다.** 이것이 지금 추출 비용을
   먼저 치르는 이유다.
