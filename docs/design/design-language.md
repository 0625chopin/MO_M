# mo_im 디자인 언어 — 잉크와 확정성

디자인 개편(2026-07-24) 산출물. 참조: **D-038**, D-026, D-011, D-029, NFR-017·018·020·021·022·023·026·027, CON-09, R-002.

이 문서는 `src/app/globals.css`의 토큰이 **왜 이 값인지**를 재현 가능하게 기록한다.
12색 팔레트 자체의 유래는 [`calendar-palette.md`](./calendar-palette.md)가 담당하며, 이 문서는
**그 팔레트를 둘러싼 나머지 전부**(뉴트럴·타이포·상태 표현·표면 규칙)를 다룬다.

## 0. 개편 전 상태와 문제

`shadcn init`이 생성한 기본값이 그대로 남아 있었다. [`../decisions/shadcn-ui-adoption.md`](../decisions/shadcn-ui-adoption.md)도
"**브랜드 컬러 아님, shadcn CLI 기본 placeholder다. DESIGN이 실제 값으로 덮어쓸 것을 전제로 한다**"라고
명시해 두었다. 실측으로 확인한 문제는 넷이다.

| # | 문제 | 근거 |
| --- | --- | --- |
| 1 | **한글 서체가 지정돼 있지 않았다** | `layout.tsx`가 Geist/Geist Mono만 로드했고 Geist에는 한글 글리프가 없다. 제품 언어는 한국어 단독(D-011)이므로 **화면의 거의 모든 문자**가 OS 폴백(Windows 맑은 고딕 / macOS Apple SD Gothic Neo)으로 렌더됐다 — 자간·굵기·행간이 기기마다 달라 조판을 통제할 수 없었다 |
| 2 | 뉴트럴이 전부 순수 무채(`oklch(x 0 0)`) | shadcn 기본 팔레트 그대로 |

## 1. 두 가지 규칙

### ① 채도는 데이터만 쓴다

화면의 유채색은 **크루 12색(`--crew-*`)이 전부 가져간다.** UI 크롬(버튼·경계·텍스트·포커스)은
채도 0.02 이하의 잉크 뉴트럴(H=258)로 물러난다.

**왜 무채가 아니라 잉크인가**: 순수 무채(`C=0`)는 shadcn 기본값의 지문이고, 넓은 면에서 화면이
탁해 보인다. H=258 청색 쪽으로 아주 조금(C≤0.022) 밀면 인상이 서늘해지면서도 크루색과 경쟁하지
않는다.

**왜 크롬이 유채색을 쓰면 안 되는가**: 이 앱에서 색은 **크루를 식별하는 데이터**다(FR-062).
크롬이 파란 버튼·초록 배지를 쓰면 사용자는 "이건 어느 크루 색이지?"를 매번 판단해야 한다.
같은 이유로 **활성 내비게이션 표시를 배경 칠에서 잉크 바로 바꿨다** — 배경 칠은 색 면적을
늘리는데, 색 면적은 크루 식별에 배정된 자원이다.

**예외 둘**: `--destructive`(파괴 동작·오류)와 크루 팔레트. destructive는 관례상 붉어야 하고,
실제 사용처에서 shadcn이 `bg-destructive/10`(연한 배경 + 붉은 글씨)을 쓰므로 화면에 큰 붉은 면이
생기지 않는다. 다만 **`--destructive`와 `--crew-12`(red `#d20000`)는 색상이 가깝다** — 크루 12번
크루의 확정 칩 옆에 삭제 버튼을 나란히 놓는 배치는 피한다.

### ② 배경·카드 표면의 밝기는 재량이 아니다

[`calendar-palette.md`](./calendar-palette.md)가 12색의 3:1 대비(NFR-018)를 라이트 `oklch(1 0 0)` /
다크 `oklch(0.145 0 0)` 배경 기준으로 검증했고, **최저치가 3.26이라 여유가 거의 없다.**
휘도 창은 `Y ∈ [0.1091, 0.3000]`으로 두 배경에 딱 맞춰져 있다.

그래서 배경을 크림색이나 짙은 남색으로 바꾸는 선택지는 **애초에 없었다.** 색조는 배경이 아니라
표면·경계·텍스트에서 낸다.

#### 표면별 실측 — 크루색을 어디에 놓아도 되는가

| 표면 | 값 | 12색 최저 대비 | 판정 |
| --- | --- | ---: | --- |
| 라이트 `--background` | `oklch(1 0 0)` | 3.26 | ✅ |
| 라이트 `--card` | `oklch(1 0 0)` | 3.26 | ✅ |
| 라이트 `--muted`(개편 전 `oklch(0.97 0 0)`) | — | **2.99** | ❌ crew-3(pink)·crew-6(periwinkle) |
| 라이트 `--muted`(개편 후) | `oklch(0.966 0.005 258)` | **2.97** | ❌ |
| 다크 `--background` | `oklch(0.145 0 0)` | 3.52 | ✅ |
| 다크 `--card` | `oklch(0.2 0.01 258)` | 3.22 | ✅ |

> **규칙: 크루색은 `--background`와 `--card` 위에만 놓는다.**
> `--muted`/`--secondary`/`--accent` 표면에 크루 칩을 올리면 3:1이 깨진다. 이건 개편이 만든
> 제약이 아니라 **원래부터 있던 함정**이며(개편 전 `oklch(0.97 0 0)`에서도 2.99), 이번에
> 실측해 규칙으로 박았다. `globals.css` 상단 주석에도 같은 내용이 있다.

라이트 `--card`를 흰색으로 유지한 것은 이 때문이다. 다크 `--card`는 `0.205`에서 `0.2`로 아주
조금 낮춰 여유를 벌었다(3.18 → 3.22).

## 2. 확정성 스케일 (시그니처)

제품 한 줄 정의는 "**채팅에 묻힌 약속을 투표로 확정해 캘린더에 남긴다**"이다. 그 서사를 채도에
실었다 — **확실할수록 색이 찬다.**

| 단계 | 의미 | 표현 |
| --- | --- | --- |
| `certainty-draft` | 제안·투표 중 | 점선 테두리 · 무채 · 배경 없음 |
| `certainty-pending` | 정족수 충족, 판정 대기 | 실선 테두리 · 크루색 12% 배경 |
| `certainty-confirmed` | 가결·확정 | 크루색 채움 |

캘린더 셀·투표 카드·Meetup 배지·알림이 **전부 같은 세 단계**를 쓴다. 상태를 색 하나로만
전달하지 않는 것도 의도다(WCAG 1.4.1) — 점선/실선/채움이라는 **형태 차이**가 색과 함께 간다.

구현은 `globals.css`의 `@utility certainty-*` 셋이고, 호출부는 `crewCertaintyVars(index)`가
돌려주는 CSS 변수 두 개를 인라인 스타일로 넘긴다.

### 채움 위 글자색은 색마다 다르다

확정 상태가 크루색을 **배경으로** 쓰는 순간 그 위 글자색이 새 변수가 된다. 12색의 상대휘도가
0.1365~0.2725에 걸쳐 있어 **한 가지 글자색으로는 본문 4.5:1(NFR-018)을 맞출 수 없다.**

- `Y ≤ 0.175` → 밝은 글자(`--crew-text-paper`)만 통과
- `Y ≥ 0.1833` → 어두운 글자(`--crew-text-ink`)만 통과
- 팔레트에 **그 사이(0.175~0.1833)에 걸친 색은 없다** — 12색이 저/중/고 3밴드로 의도적으로
  분산됐기 때문이다(`calendar-palette.md` §2)

| 글자색 | 대상 크루 | 최악 대비 |
| --- | --- | ---: |
| `--crew-text-paper` | 2(violet)·4(teal)·5(green)·12(red) | 5.36 |
| `--crew-text-ink` | 나머지 8색 | 4.68 |

색마다 어느 쪽을 쓸지는 `src/lib/crew-palette.ts`의 **`textOn` 필드가 데이터로 들고 있다.**
런타임 계산이 아니라 데이터인 이유: 채움색과 글자색은 **짝**이고, 짝을 깨는 실수는 라이트 모드
테스트만으로는 드러나지 않는다. `crewCertaintyVars()`가 항상 짝으로 돌려주는 것도 같은 이유다.

크루색이 라이트·다크 단일값이므로(D-026) **이 두 글자색도 단일값이다** — `.dark`에서 재정의하지
않는다.

## 3. 타이포그래피

| 역할 | 서체 | 설정 |
| --- | --- | --- |
| 본문·제목 | **Noto Sans KR** | **가변**(wght 100~900) |
| 수·시각 | **Geist Mono** | `tabular-nums`(`.tnum` 유틸) |

**왜 가변인가** — 한글 웹폰트에서 이건 취향이 아니라 용량 문제다. 구글 CSS API 실측:

| 설정 | 조각 수 | 총 용량 |
| --- | ---: | ---: |
| 고정 400 + 600 | 248 | 6,874 KB |
| **가변 100~900** | **124** | **3,437 KB** |

가변 쪽이 **조각 수도 용량도 절반**이면서 굵기 전 범위를 준다. 고정 굵기를 하나만 써도
124조각·3.4MB로 가변과 같으므로, 이 폰트에서 가변은 언제나 이득이다.

제목은 별도 디스플레이 서체를 더하지 않고 **600 + 음수 트래킹(-0.022em)** 으로 세운다. 위계는
서체 수가 아니라 아래 mono의 역할 분담이 만든다.

> **개편 이력**: 처음에는 IBM Plex Sans KR(고정 400·600)로 갔다가 **Noto Sans KR 가변으로
> 교체**했다(2026-07-24, 고객 지시). IBM Plex Sans KR은 라틴 IBM Plex와 한 가족이라 라틴
> 워드마크와의 조화가 좋았지만 **가변 판이 없어** 굵기마다 별도 글리프 한 벌이 필요했다.
> 교체 후 실측에서 한 페이지 기준 실제 로드 조각이 **30개 → 20개**로 줄었다(굵기별 중복
> 로드가 사라졌다).

**모노의 역할**: 코드 블록용이 아니라 **수·시각의 서체**다. 이 앱은 득표수·정족수·시각·날짜가
화면의 뼈대라(FR-020~025, FR-062) 숫자에 고유한 얼굴을 주는 것이 위계 장치가 된다.

> **모노에 한글을 넣지 않는다.** Geist Mono에는 한글 글리프가 없어 OS 폰트로 폴백되고, 한 줄
> 안에서 서체가 갈린다("8월 14일"의 숫자와 한글이 서로 다른 얼굴이 된다). 한글이 섞이면
> **sans + `.tnum`** 을 쓴다 — 자릿수 정렬은 그것만으로 얻어지고 서체는 하나로 남는다.
> 개편 중 `/sample`과 `PageHeader.eyebrow`에서 실제로 이 실수를 하고 화면에서 발견해 고쳤다.

### 한국어 조판

`html`에 `word-break: keep-all` + `overflow-wrap: break-word`를 건다. 이게 없으면 브라우저가
한글을 **음절 단위로** 끊어 "일정을 확 / 정합니다"처럼 낱말 한가운데서 줄을 바꾼다. 360px
뷰포트(NFR-026)에서 특히 자주 나온다. 본문 행간은 `--leading-body: 1.7` — 한글은 라틴보다
글자틀이 꽉 차 같은 행간에서 더 답답해 보인다.

### 폰트 로딩 — `preload: false`는 의도적이다

구글은 한글 폰트를 유니코드 범위별로 잘게 쪼개 배포하고 next/font는 그 조각을 전부 셀프
호스팅한다(Noto Sans KR 가변 기준 빌드 산출물 **130개 woff2, 3.8MB**).

| 설정 | 초기 `<link rel="preload">` | 실제 로드 |
| --- | ---: | ---: |
| `preload: true`(기본) | 수십 건 (IBM Plex 시절 실측 **47건**) | — |
| `preload: false`(채택) | **0건** | **20개** (브라우저가 `unicode-range`로 선택) |

preload를 켜면 그 페이지에 실제로 쓰인 글자가 속한 조각이 몇 개뿐인데도 수십 건이 한꺼번에
나간다. 끄면 브라우저가 화면에 나온 글자가 속한 조각만 받는다. 대가는 첫 페인트의 폴백 서체
노출인데(`display: "swap"`), next/font가 폴백 메트릭을 맞춰 주므로 레이아웃 이동(CLS)은 없다.

## 4. 그 밖의 토큰 변경

| 토큰 | 개편 전 | 개편 후 | 근거 |
| --- | --- | --- | --- |
| `--radius` | `0.625rem` | `0.5rem` | 캘린더 셀·크루 칩·투표 행처럼 작고 촘촘한 요소가 주역이라 기본값의 둥근 모서리는 밀도를 흐린다. 파생 반경이 전부 이 값의 배수라 하나만 바꾸면 전체가 따라온다 |
| `--ring` | `oklch(0.708 0 0)` — **2.59, 미달** | 라이트 `oklch(0.45 0.02 258)` 7.43 / 다크 `oklch(0.72 0.025 258)` | WCAG 2.2 §2.4.11·NFR-020. 라이트·다크가 배경이 뒤집히므로 값을 따로 준다. 크루 파랑(crew-9)과 혼동되지 않게 채도는 0.02로 묶고 명도로만 드러낸다 |
| `--muted-foreground` | `oklch(0.556 0 0)` — 4.73 | `oklch(0.53 0.016 258)` — 5.28 | 보조 문구도 본문 기준 4.5:1을 지켜야 한다 |
| `--destructive` | `oklch(0.577 0.245 27.325)` — 4.76 | `oklch(0.52 0.21 27)` — 6.15 | `text-destructive`로 본문에 직접 쓰이는 색이라 여유를 뒀다 |
| `--chart-1`~`5` | 무채 그레이스케일 | 잉크 스케일 | 차트는 v0.1 화면에 없지만 표준 토큰이라 맞춰 둔다. **차트가 생기면 크루색이 아니라 이 스케일을 쓴다** — 크루색은 크루 식별 전용이다 |

### 다크 블록 두 개는 값이 같아야 한다

명시적 테마 토글(ThemeProvider)이 아직 없어(**I-020**) `globals.css`는 다크 값을 두 곳에 쓴다 —
`@media (prefers-color-scheme: dark)` 폴백과 `.dark` 클래스. **한쪽만 고치면 OS 다크와 명시 토글이
서로 다른 화면을 그린다.** 개편 후 두 블록이 31개 선언에서 완전히 일치함을, 그리고 `--crew-*`가
폴백 블록에 없음을(D-026) 파싱해 확인했다.

## 6. 검증 기록 (2026-07-24)

- `npx tsc --noEmit` 통과 · `npm run lint` 오류/경고 0 · `npm run build` 통과(15개 라우트)
- 대비 수치는 전부 **직접 계산**했다(OKLCH → 선형 sRGB → WCAG 상대휘도). 계산 근거는 이 문서의
  표에 그대로 있고, 팔레트 쪽 수치는 `calendar-palette.md`와 `crew-palette.ts`가 이미 들고 있다
- 브라우저 실측(Playwright): 라이트·다크 렌더, 360px에서 가로 스크롤 없음(`scrollWidth == clientWidth`),
  `word-break: keep-all` 적용, `certainty-*` 3단계 렌더, 프로덕션에서 폰트 preload 0건·실제 로드 20건
  (Noto Sans KR 가변 교체 후 재측정)
- 다크 블록 동기화: `@media` 31선언 == `.dark` 31선언, `--crew-*` 폴백 미포함

## 7. 이 문서를 고쳐야 하는 때

- `--background`·`--card`의 **밝기**를 바꾸려 할 때 → 먼저 `calendar-palette.md`의 휘도 창을
  다시 계산한다. 12색 전체가 무효화될 수 있다
- 크루 팔레트에 색을 추가·변경할 때 → `textOn`·`contrastOnFill`을 함께 재계산한다
- 폰트를 바꾸거나 굵기를 추가할 때 → §3의 조각 수·preload 실측을 다시 잰다
