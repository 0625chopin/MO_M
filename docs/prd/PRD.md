## 1. 핵심 정보

**목적**: 동호회(크루)의 모임 일정이 채팅에 묻혀 기록이 남지 않고, 확정된 일정을 각자 개인 캘린더에 옮겨 적어야 하는 문제를 없앤다 — 게시글 기반 찬반 투표로 일정을 확정하고, 가결된 일정을 크루 색으로 구분된 통합 캘린더에 자동 등록한다.

**사용자**: 동호회·소모임을 운영하거나 참여하는 일반 성인 사용자 — 크루를 만들고 운영하는 **오너/임원**, 투표하고 참석 여부를 밝히는 **크루원**, 새 크루를 찾는 **일반회원**.

## 2. 역할과 권한 요약

원문 №4·№12와 D-002·D-007이 **권한 모델 자체를 제품 기능**으로 만들었으므로, 구현 수준 보안 설계(RLS 정책 문구·토큰 저장 방식 등, `requirements.md` 5.3절 소관)는 다루지 않고 **역할별로 무엇에 접근·행동할 수 있는가**만 요약한다.

### 2.1 역할 정의 (`requirements.md` 3.1절)

| 역할 | 식별자 | 범위 | 설명 |
| --- | --- | --- | --- |
| 비회원 | `guest` | 전역 | 미인증 방문자 |
| 일반회원 | `member` | 전역 | 인증됐으나 해당 크루에 비소속 |
| 크루원 | `crew_member` | 크루별 | 해당 크루 소속(`status=active`) |
| 임원 | `crew_staff` | 크루별 | 크루원 + 운영 권한(가입 승인, 초대, 게시글 관리) |
| 크루 오너 | `crew_owner` | 크루별 | 크루당 **정확히 1명**. 임원 권한 전부 + 크루 존폐 권한 |
| 시스템 관리자 | `system_admin` | 전역 | 신고 처리(v0.2 이후 본격 사용) |

**한 사용자가 크루 A의 오너이면서 크루 B의 일반 크루원일 수 있다** — 역할은 크루 단위로 부여된다.

## 3. 사용자 여정

### 여정 A — 신규 가입 → 크루 탐색 → 가입 신청 → 승인

```
1. 랜딩 페이지
   ↓ "시작하기" 클릭
2. 회원가입 페이지 (F001)
   ↓ 이메일 인증 완료
3. 온보딩 페이지 (F003 — 핸들·표시이름 설정)
   ↓ 완료
4. 홈 대시보드 페이지 (소속 크루 0개 → 크루 탐색 유도)
   ↓ "크루 탐색" 클릭
5. 크루 탐색 페이지 (F008)
   ↓ 카테고리/키워드로 원하는 크루 발견 → 상세 진입
6. 크루 홈 페이지 (F006)
   ↓ "가입 신청" 클릭 (F011)
   [조건: public 크루] → 신청 접수, 오너·임원 전원에게 알림 → 대기
   [조건: private 크루] → 가입 신청 버튼 비노출, "초대 전용" 안내만 표시
   ↓
7. 멤버 관리 페이지 — 임원/오너가 신청 확인 후 승인 (F012)
   ↓ 승인
8. 크루 홈 페이지 재진입 → 게시판·채팅 즉시 이용 가능
```

### 여정 B — 크루 개설 → 크루원 초대 → 임원 임명

```
1. 홈 대시보드 페이지
   ↓ "크루 만들기" 클릭
2. 크루 개설 페이지 (F005)
   ↓ 크루명·소개·카테고리·공개범위·색 입력 → 제출
3. 크루 홈 페이지 — 즉시 생성(D-008, 승인 대기 없음). 게시판·채팅방 자동 생성
   ↓ "멤버 관리" 진입
4. 멤버 관리 페이지 (F009 — 핸들 검색 F004 후 초대)
   ↓ 초대 발송
   [대상 수락] → 받은 초대함 페이지(F010) → 수락 → 크루원(active)으로 전환
   [대상 거절] → declined, 재초대 가능
   ↓
5. 멤버 관리 페이지 — 오너가 신뢰하는 크루원을 임원으로 임명 (F013)
   ↓ 임명 완료
6. 해당 크루원이 멤버 관리 페이지에서 가입 승인·강퇴 등 운영 권한 행사 가능
```

### 여정 C — 모임 제안 → 투표 → 가결 → 캘린더 → 참석 신청

```
1. 커뮤니티 게시판 페이지
   ↓ "글쓰기" 클릭
2. 글쓰기 페이지 — "모임 제안" 유형 선택 (F019)
   ↓ 제목·설명·모임 예정일·투표 마감(+선택: 시각·장소·정원) 입력 → 등록
3. 게시글 상세 페이지 — 게시글 + 투표(Poll, open) 동시 생성 (F020)
   ↓ 크루원 알림 발송
4. 게시글 상세 페이지 — 크루원이 찬성/반대/기권 투표 (F021), 실시간 현황 확인 (F022)
   ↓ 마감 도래 / 조기 종료 / 전원 투표 완료 (F023)
5. 투표 결과 판정 (F024)
   [가결: 정족수 1/3 충족 AND 찬성>반대] → Meetup 자동 생성 (F030)
   [부결: 동수 포함] / [무효: 정족수 미달] → Meetup 생성 안 됨
   ↓
6. 알림 센터 페이지 + 토스트 — 투표 대상자에게 결과 통지 (F025). 탈퇴자·강퇴자 제외(D-015)
   ↓ (가결 시)
7. 통합 캘린더 페이지 — 크루 색 바로 표시 (F031, F032)
   ↓ 날짜 클릭 (조회 전용, D-012)
8. 통합 캘린더 페이지 — 날짜 상세 패널에서 해당일 Meetup 목록 표시 (F033)
   ↓ 항목 클릭
9. Meetup 상세 페이지 (F034)
   ↓ 정원이 지정된 경우
   [여유 있음] → "참석" 클릭 → 참석 확정 (F035)
   [정원 마감] → 참석 버튼 비활성, "마감되었습니다" 안내
   ↓ (참석 취소 시)
10. 참석 취소 → 자리 즉시 반환, 다른 크루원이 신청 가능 (F036). 대기열은 없음(D-013)
```

## 8. 기술 스택

**설치된 것과 도입 예정을 구분해 적는다**(D-036). 한 목록에 섞으면 전부 준비된 것처럼 읽혀 착수 시 잘못된 전제가 된다. 항목을 추가할 때 어느 구획인지 밝히지 않으면 등재하지 않는다.

### 8.1 이미 설치됨 (`package.json` 실측)

런타임 의존성은 **세 개뿐**이다.

- **Next.js 16.2.11** — App Router, Turbopack이 dev·build 기본 번들러, React Compiler 활성화(`reactCompiler: true`)
- **React 19.2.4** / **react-dom 19.2.4**
- **TypeScript**(strict, 개발 의존성) — 경로 별칭 `@/*` → `./src/*`
- **Tailwind CSS v4**(개발 의존성) — CSS-first, `tailwind.config.*` 없음. 토큰은 `src/app/globals.css`의 `@theme inline`

> **`globals.css`에 캘린더 12색 팔레트는 아직 없다.** 현재 토큰은 background/foreground/폰트뿐이다(CON-04 대상, D-026으로 색값 결정 방식만 확정).

### 8.2 도입 예정 (아직 의존성에 없음)

| 항목 | 용도 | 시점 | 근거 |
| --- | --- | --- | --- |
| **Supabase JS 클라이언트** | 인증·DB·Realtime | v0.2 | CON-01 |
| **shadcn/ui** | 컴포넌트 기반 (MCP는 이미 설정됨) | v0.1 | 공통 기반 48인일 중 10~15인일 단축 여지 |
| **커스텀 SMTP 공급자** | 인증 메일 발송 | v0.2 | **D-021** — 공급자 미정(**I-016**) |

### 8.3 백엔드·인프라

- **Supabase** — 인증, PostgreSQL, **Realtime Broadcast**(D-023)
  - **project ref `damruradpliktkrlkakl` — 프로젝트명 `MO-IM`, mo_im 전용.** 타 앱(축구 매니저 시뮬레이션)이 점유하던 것을 **초기화해 재사용**했다(**D-037**). 테이블·함수·`auth.users`·마이그레이션 이력이 모두 비어 있음을 확인했다. **ref 교체는 하지 않는다** — D-018의 전용 프로젝트 요건은 초기화로 충족했고 **R-018은 종결**됐다.
  - 이전 앱의 Edge Function 2개도 제거해 **완전히 빈 프로젝트**임을 확인했다(테이블·함수·`auth.users`·마이그레이션·Storage·Edge Function 모두 0).
  - **요금제 미확정**(**I-017**) — NFR-006의 1,000세션은 Pro + 지출 상한 해제 이상을 전제한다(Free 200 / Pro 500).
- **스케줄 실행: Supabase Cron(pg_cron)** — 투표 자동 종료(D-003)와 채팅 12개월 파기(D-009)에 쓴다. **Vercel Cron은 쓰지 않는다**(Hobby는 하루 1회·±59분이며 더 잦은 표현식은 배포가 실패한다). 동시 잡 8개·잡당 10분 이내 (**D-027**)
- **Vercel** — 배포 (스케줄 용도로는 쓰지 않는다)

### 8.4 개발 방법론

- **Mock First**: 모든 컴포넌트는 `/sample` 쇼케이스(4상태: 정상/로딩/빈 상태/오류)에 먼저 등록하고, Mock 데이터로 화면을 완성한 뒤 Supabase를 연결한다. Mock과 실데이터는 동일 TypeScript 타입을 공유한다(7절).
- **전환 경계 4개를 v0.1부터 지킨다**(**D-030**) — ① 표현/컨테이너 분리(Mock에도 컨테이너를 만든다) ② `subscribeToRoom(id, onEvent): Unsubscribe` 구독 인터페이스 추상화 ③ 4상태 "오류"에 **도메인 오류**(RLS 403·정원 마감·동시 수정 충돌) 포함 ④ 인증 경계는 **레이아웃**에서 처리(`proxy.ts`는 D-011로 범위 밖). Server Action + `refresh()` 패턴을 Mock 단계부터 쓴다.
- **React Compiler**: 수동 `useMemo`/`useCallback`/`memo`를 쓰지 않되, **측정 근거를 기록하면 `"use no memo"` 예외를 허용한다**(**D-029**). INP 목표는 메모이제이션이 아니라 **렌더링 전략**(윈도잉·`startTransition`·안정적인 `key`)으로 달성한다.

## 11. 정합성 검증 체크리스트 실행 결과

### 1단계 — 기능 명세 → 페이지 연결

- F001~F039 각각의 "관련 페이지" 열에 적힌 페이지 이름을 6절 페이지별 상세 기능 목록과 하나씩 대조했다. **39건 전부 6절에 동일 이름의 페이지가 존재함을 확인.** (예: F028 게시글 링크 공유 → "게시글 상세 페이지, 채팅방 페이지" 모두 6절에 존재)
- 최초 초안에서 F035(참석/불참 응답)를 Meetup 상세 페이지에만 연결했으나, FR-066 원문 정상 흐름에 "①Meetup 상세 **또는 캘린더 패널** 진입"이 명시돼 있어 **통합 캘린더 페이지도 관련 페이지에 추가**했다(수정 완료).

### 2단계 — 메뉴 구조 → 페이지 연결

- 5절 메뉴 구조의 모든 항목(홈 대시보드, 크루 탐색, 통합 캘린더, 알림, 계정 설정, 받은 초대함, 크루 개설, 크루 홈, 커뮤니티 게시판, 글쓰기, 게시글 상세, 채팅, 멤버 관리, 크루 설정, Meetup 상세, 회원가입, 로그인, 로그아웃)를 6절 페이지 목록과 대조 — **전부 존재 확인.**
- 메뉴에 표기된 F-ID(F001~F039 중 사용된 것)를 4절 기능 명세 표와 대조 — **전부 정의됨을 확인.** 로그아웃은 F002(로그인·로그아웃·세션 유지)에 이미 포함된 기능이라 별도 F-ID를 만들지 않았다(중복 방지).

### 3단계 — 페이지별 상세 기능 → 역참조

- 6절 19개 페이지의 "구현 기능 ID" 전부를 4절 표에서 재조회 — **전부 존재.**
- 5절 메뉴 구조를 훑어 6절의 19개 페이지 전부가 최소 한 경로 이상으로 도달 가능한지 확인 — **전부 도달 가능.** (Meetup 상세 페이지는 헤더 메뉴에는 없으나 통합 캘린더의 날짜 패널을 통해 도달하며, 이는 5절 "캘린더 하위" 항목으로 명시했다.)

### 4단계 — 누락·고아 항목

- **기능 명세에만 있고 페이지에 없는 기능**: 없음. F001~F039 전부 최소 1개 페이지에 등장.
- **페이지에만 있고 기능 명세에 없는 기능**: 없음. 6절의 모든 "구현 기능 ID"가 4절 표에 존재.
- **메뉴에만 있고 페이지가 없는 항목**: 없음. 단, 초안에서 메뉴 구조에 "온보딩" 항목을 상시 메뉴로 넣었다가 — 온보딩은 재방문 시 홈으로 리다이렉트되는 1회성 화면이라 상시 내비게이션 항목이 아님을 확인하고 **메뉴에서 제거**, 6절 페이지 설명에만 "진입 경로: 회원가입 후 자동 진입"으로 남겼다(수정 완료).

### 부가 검증 — F ↔ FR 추적

- 9.1절(F→FR)과 9.2절(FR→F)을 상호 대조해 39쌍이 양방향으로 정확히 일치하는지 확인 — **일치.**
- 9.3절의 v0.1 범위 밖 15건과 4.2절 표를 대조 — **동일 15건, 누락 없음.**
- 39 + 15 = 54가 `prioritization-and-risks.md` 6.1절 FR 총계(36+14+3+1)와 일치함을 재계산으로 확인.

**결론: 4단계 체크리스트 전 항목 통과. 발견된 2건(F035 페이지 누락, 온보딩 메뉴 오분류)은 모두 수정 완료.**

## 12. 원본 요구사항 문서에서 발견한 모순·누락·모호점

> **전건 처리 완료 (2026-07-23).** 1·2번은 원본 문서의 집계 오류라 그대로 수정했고, 3·4·5번은 고객 확인을 거쳐 **D-015·D-016·D-017**로 확정했다. 아래 기술은 발견 당시의 내용이며, 각 항목 끝에 처리 결과를 붙였다.

1. **`requirements.md` 부록 A "FR 인덱스"에 FR-066~FR-068이 누락됨.** 부록 A는 FR-001부터 FR-082까지 나열하며 "합계: FR 51건"이라고 적었으나, 실제로 세어보면 FR-066(참석/불참 응답)·FR-067(참석 취소 시 자리 반환)·FR-068(참석자 목록 조회) **3건이 목록에서 빠져 있다.** 이 3건은 4.G절 본문에는 정상적으로 정의돼 있고 D-013 결정으로 신규 추가된 것으로 5.5절·6.1절(prioritization-and-risks.md)에도 명확히 반영돼 있는데, 부록 A만 D-013 반영 이전 상태로 남아 있다. `prioritization-and-risks.md` 6.1절은 이 3건을 포함해 "총 54건"으로 정확히 집계했으므로, **두 문서 간 FR 총계가 51 vs 54로 어긋난다.** (이 PRD는 54건 기준으로 작성했다.) → **처리: 부록 A에 FR-066~068 3행을 추가하고 합계를 54건(원문 25 / 보완 29)으로 정정했다.**
2. **엔티티 개수 표기가 문서 내에서 3가지로 다르다.** `requirements.md` 5.5절 D-001 결정문은 "엔티티 17종", 같은 문서 6.2절(prioritization-and-risks.md에 인용된) R-010 리스크 설명은 "엔티티 16종"이라 하는데, 실제 5.2절 데이터 모델 표에는 `DevicePushToken`(차기 테이블)까지 포함해 **20개 엔티티**가 나열돼 있다. `MeetupAttendance`(D-013)가 나중에 추가되며 개수 표기가 갱신되지 않은 것으로 보이나, 16·17·20 중 무엇이 맞는 숫자인지 원본만으로는 확정할 수 없다. → **처리: 실제 표를 기준으로 20종으로 통일했다**(`DevicePushToken`은 차기용 타입 자리). 5.2절 머리말에 명시하고 R-010·D-001의 서술도 맞췄다.
3. **FR-045(투표 종료 알림) 대상자 정의가 FR-041(투표 참여) 대상자 정의와 미묘하게 다르다.** FR-045는 "투표 대상자 **전원**(투표 여부 무관)과 제안자"에게 알림을 보낸다고 하는데, 3.4절 "투표 중 탈퇴자" 규칙은 강퇴자의 표를 "무효화하고 분모에서도 제외"한다고만 하고 **강퇴된 사용자가 알림 대상에 남는지(FR-045)는 명시하지 않는다.** FR-045 예외흐름 E3은 "탈퇴자는 발송 제외"라고만 하고 **강퇴자**에 대해서는 언급이 없어, 강퇴된 사용자에게 투표 결과 알림이 가는지 애매하다. → **처리: D-015 — 강퇴자에게는 발송하지 않는다.** FR-045 E3을 "탈퇴자·강퇴자 제외"로 확장했다. 표의 유효성 판정(탈퇴자 유효/강퇴자 무효)은 D-003 그대로다.
4. **크루 개설 시 색상 입력(FR-010 정상 흐름 "색상 입력")과 D-006의 "자동 배정이 기본값"이 순서상 충돌한다.** FR-010은 크루 개설 화면에서 "크루명·소개·카테고리·공개 범위·**색상** 입력"을 정상 흐름에 넣었는데, D-006/FR-062는 "오너가 팔레트 내에서 직접 고를 수 있게 한다(**자동 배정이 기본값**)"이라 해 기본은 자동이어야 한다. 개설 화면에 색상 선택이 필수 입력처럼 보이는지, 아니면 자동 배정된 값을 개설 화면에서 바꿀 수만 있는 것인지가 원문에서 명확히 구분되지 않는다. → **처리: D-016 — 개설 시 자동 배정하고 개설 폼에서 색을 묻지 않는다.** 변경은 크루 설정(SC-15)에서만 한다.
5. **FR-012(공개 범위)의 AC1과 FR-014(크루 검색)의 대상 범위가 "검색"과 "목록 열람"을 구분 없이 섞어 쓴다.** FR-012 AC1은 "`private` 크루, 비소속 회원이 크루 **검색**, 결과에 나타나지 않는다"고 하는 반면, 3.3절 권한 매트릭스는 "크루 검색·목록 열람"을 FR-014 한 행으로 묶어 두었다. `private` 크루가 "전체 크루 목록"(카테고리 브라우징처럼 검색어 없이 훑어보는 화면)에도 나타나지 않는지, 아니면 검색어를 입력했을 때만 제외되는지는 원문상 구분되지 않는다(실무적으로는 둘 다 제외가 맞아 보이나 원문이 명시하지 않음). → **처리: D-017 — 모든 조회 경로에서 제외한다.** 화면 단위 필터가 아니라 데이터 접근 규칙(RLS)으로 구현한다.
