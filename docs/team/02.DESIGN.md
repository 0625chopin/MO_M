# 02. DESIGN

**B팀** 소속이며 주 담당은 **디자인 시스템·접근성**이다. 12색 팔레트(**CON-04**·**D-026**)부터 원자 컴포넌트 15종, `/sample` 4상태 셸(**CON-09**), 마지막 접근성 QA 패스까지 **한 사람이 기준을 세우고 그 기준으로 검사**해야 기준이 흔들리지 않는다. **접근성과 4상태는 나중에 소급 적용하는 비용이 처음부터 지키는 비용보다 훨씬 크다**(**R-020**).

## 배정 근거

- **팔레트를 정의한 사람이 팔레트를 쓰는 화면을 만든다.** **D-026**은 색각 이상(2형) 시뮬레이션 후 ΔE 최대화 순으로 팔레트 순서를 정하고, 같은 날짜 셀 충돌 회피를 **인덱스 인접이 아니라 지각적 거리**로 하라고 요구한다. 이 규칙이 실제로 지켜지는지는 캘린더 MeetupBar를 만들 때 드러난다 — 그래서 캘린더를 보조로 겸임한다
- **접근성 기준을 만든 사람이 검사한다.** 원자 컴포넌트의 포커스 트랩·Esc·복귀 포커스(**NFR-020**)와 터치 대상(**NFR-027**)·대비(**NFR-018**)를 정의한 사람이 19개 페이지 × 라이트/다크 × 3뷰포트 접근성 QA를 맡으면, 자기가 정한 기준을 자기가 어긴 곳을 찾게 된다. **자동화가 없으므로 수동이다**(**R-002**)
- **`/sample` 등록 강제 규칙의 소유자다.** 테스트 러너가 없는 동안 `/sample`이 유일한 회귀 확인 지점이며(**R-002**), 실물과 쇼케이스가 어긋나면 **R-006**이 재발한 것이다. 규칙을 만든 사람이 대조도 한다
- **이 사람에게 몰리면 안 되는 것**: `🚧` Task를 주지 않는다. 크리티컬 패스 위의 Task를 가장 많이 지고 있어 **차단 Task까지 얹으면 지연이 곱해진다**
- **주의**: 초반에 앱 셸 → `/sample` 셸 → 원자 컴포넌트를 혼자 연속으로 지고, 그 셋이 **화면 Task 전량의 선행**이다. 이 구간에서 멈추면 나머지 3명이 전원 대기하므로 **진행이 막히면 즉시 알린다.** 후반에도 읽기·쓰기 교체와 통합 테스트로 크리티컬 패스 위에 있다
