# ISSUES

개발 중 발견한 **미결 이슈와 개선사항**을 기록한다. 이 파일이 이슈 번호의 **단일 소스**다.

- **다음 이슈 번호: I-016** (등재할 때마다 이 줄을 갱신한다)
- 확정된 **결정**은 여기가 아니라 [`prioritization-and-risks.md`](./prioritization-and-risks.md) 6.3절 결정 기록(D-\*)에 쓴다. 결정과 미결을 같은 곳에 두지 않는다.
- 이슈는 **누구나 제보**한다. 등재할 때 형식(아래 "기록 형식")을 지키고 "다음 이슈 번호" 줄을 함께 갱신한다.

## 기록 형식

```md
### I-001 · 제목 한 줄

- **상태**: 열림 | 진행중 | 해결됨 | 보류 | 무효
- **영역**: 예) UI / 데이터 / 빌드 / 접근성 / 문서
- **제보**: 이름 (YYYY-MM-DD)
- **내용**: 무엇이 문제인가. 재현 경로나 해당 파일 경로를 포함한다.
- **영향**: 방치했을 때 무엇이 깨지거나 느려지는가.
- **후속**: 해결 시 커밋/PR 링크, 결정으로 승격됐다면 D-\* 번호.
```

상태를 `해결됨`·`무효`로 바꿀 때도 항목을 지우지 않는다. 같은 문제가 다시 올라왔을 때 이전 판단을 찾을 수 있어야 한다.

### I-014 · Meetup에 시각·장소·정원 필드가 필요한지 미정이다

- **영역**: 요구사항
- **제보**: requirements-analysis-expert (2026-07-23)
- **내용**: 원문 №9·№10은 날짜만 언급한다. 실제 동호회 일정은 보통 시각·장소가 필요하고, 정원이 있으면 참석 확정 기능(별도 요구사항)이 따라온다. `requirements.md` FR-034는 필수 입력을 날짜까지로 두고 나머지를 선택·미결로 표시했다.

- **후속**: 2026-07-23 고객 검토에서 날짜 필수 + 시각·장소·정원 선택, 정원 지정 시 선착순 참석 신청·대기열 없음으로 확정 → **D-013**(`prioritization-and-risks.md` 6.3절). 결정 내용·대안·영향은 그쪽을 단일 소스로 본다.
