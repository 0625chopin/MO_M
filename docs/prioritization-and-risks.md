# 우선순위와 리스크

요구사항의 **우선순위**, **리스크 등록부**, **확정된 결정**을 기록한다.

- 이 파일의 6.3절 **결정 기록(D-\*)**이 확정 결정의 **단일 소스**다. **미결 이슈·개선사항은 여기가 아니라 [`ISSUES.md`](./ISSUES.md)**에 쓴다. 결정과 미결을 같은 곳에 두지 않는다.
- **다음 결정 번호: D-015** / **다음 리스크 번호: R-018** (등재할 때마다 이 줄을 갱신한다)
- 리스크는 **누구나 제보**한다. 미결이 확정되면 `ISSUES.md`의 I-\* 를 닫고 여기 6.3절에 D-\* 로 옮긴다.
- 항목은 상태가 바뀌어도 **지우지 않는다**. 같은 판단이 다시 올라왔을 때 이전 근거를 찾을 수 있어야 한다.

> **절 번호(6.x)는 요구사항 문서 묶음에서의 위치를 따른다.** 앞선 1~5절은 [`requirements/requirements.md`](./requirements/requirements.md)가 담당한다(2026-07-23 작성). 6.1절은 그 문서의 FR-\*/NFR-\* 를 근거로 채워졌다(R-008 참고).

## 6.1 우선순위

### 기록 형식

각 요구사항에 **MoSCoW 등급**과 **릴리스 목표**를 붙인다.

| 등급 | 뜻 | 판정 기준 |
| --- | --- | --- |
| **M** (Must) | 없으면 릴리스 불가 | 빠지면 제품이 목적을 달성하지 못한다 |
| **S** (Should) | 중요하나 우회 가능 | 수동 절차나 임시 대안으로 버틸 수 있다 |
| **C** (Could) | 여유가 있으면 | 빠져도 사용자가 목적을 달성한다 |
| **W** (Won't) | 이번 범위 밖 | 다음 릴리스로 명시적으로 미룬 것 |

```md
| ID | 요구사항 | 등급 | 릴리스 | 근거 | 의존 |
| --- | --- | --- | --- | --- | --- |
| FR-001 | 한 줄 요약 | M | v0.1 | 왜 이 등급인가 | FR-000, D-001 |
```

- **근거 없는 등급은 등재하지 않는다.** "중요해 보여서"는 근거가 아니다.
- 등급을 바꿀 때는 6.3절에 D-\* 결정을 남기고 여기서 그 번호를 참조한다.
- `W` 항목도 지우지 않는다. 무엇을 **의도적으로** 안 하기로 했는지가 범위 관리의 절반이다.

### R-001 · Next.js 16의 breaking change가 학습된 관습과 충돌한다

- **상태**: 열림
- **영역**: 기술
- **확률**: 상 · **영향**: 중
- **내용**: `next@16.2.11`은 이전 메이저와 API·규약이 다르다. 특히 요청 API 비동기화(`params`·`searchParams`·`cookies()`·`headers()`·`draftMode()`), `middleware.ts` → `proxy.ts` 대체, Turbopack 기본화 및 설정 위치(`next.config.ts` 최상위 `turbopack`), 캐싱 API 변경(`revalidateTag` 동작·`updateTag`·`refresh`·`cacheLife`/`cacheTag`), `next/image` 기본값 변경, 병렬 라우트 `default.js` 필수화. 관습대로 작성한 코드가 **런타임에서야** 깨질 수 있다.
- **대응**: 완화 — 코드 작성 전 `node_modules/next/dist/docs/`의 버전 대응 문서를 읽는다(`AGENTS.md` 규약). 업그레이드 가이드는 `node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md`.
- **신호**: 리뷰에서 동기 `params` 접근, `middleware.ts` 신규 생성, `experimental.turbo` 설정이 발견됨.

### R-002 · 테스트 러너·포매터·CI가 없어 회귀를 잡을 수단이 없다

- **상태**: 열림
- **영역**: 기술
- **확률**: 상 · **영향**: 상
- **내용**: `package.json`의 스크립트는 `dev`·`build`·`start`·`lint` 뿐이다. 자동 검증 수단은 사실상 **빌드 중 타입 검사**와 ESLint 뿐이고(`npx tsc --noEmit`로 단독 실행 가능), 동작 회귀를 잡는 층이 전혀 없다. 컴포넌트가 늘어날수록 수동 확인 비용이 선형으로 증가한다.
- **대응**: 완화 — 컴포넌트가 늘기 전에 테스트 러너 도입 여부를 D-\* 로 결정한다. 도입 전까지는 `/sample` 쇼케이스가 유일한 회귀 확인 지점이므로 신규 컴포넌트 등록을 강제한다.
- **신호**: 같은 컴포넌트에서 이미 고쳤던 버그가 재발한다. 리뷰가 "로컬에서 확인했다"에 의존한다.

### R-003 · Supabase가 설정만 있고 연결되지 않아 Mock↔실데이터 전환 비용이 미지수다

- **상태**: 열림
- **영역**: 데이터
- **확률**: 중 · **영향**: 상
- **내용**: `.mcp.json`에 project ref(`damruradpliktkrlkakl`)만 있고 클라이언트 코드·스키마·마이그레이션이 없다. 개발 원칙은 "UI 컴포넌트 수정 없이 데이터 조회 부분만 교체"를 요구하지만, 그 경계(데이터 접근 레이어)가 아직 존재하지 않는다. 경계 없이 컴포넌트가 늘면 전환 시점에 **UI를 광범위하게 고쳐야 한다.**
- **대응**: 완화 — Mock과 실데이터가 **동일한 TypeScript 타입**을 쓰도록 타입을 먼저 정의하고, 조회 함수를 컴포넌트 밖으로 분리한다. 스키마 확정 후에는 `mcp__supabase__generate_typescript_types`로 타입을 생성해 수기 타입과 대조한다.
- **신호**: 컴포넌트 안에서 Mock 배열을 직접 import하거나, Mock 전용 필드가 타입에 섞인다.

### R-004 · React Compiler와 수동 메모이제이션이 충돌한다

- **상태**: 열림
- **영역**: 기술
- **확률**: 중 · **영향**: 중
- **내용**: `next.config.ts`에 `reactCompiler: true`가 켜져 있고 `babel-plugin-react-compiler@1.0.0`을 쓴다. 습관적으로 넣은 `useMemo`/`useCallback`/`memo`는 불필요하며 컴파일러 최적화와 충돌할 수 있다. 또한 Babel 경유 변환이라 빌드 시간에 영향을 준다.
- **대응**: 수용 + 완화 — 수동 메모이제이션을 금지 관습으로 두고 리뷰에서 걸러낸다. 빌드 시간이 문제가 되면 그때 D-\* 로 재검토한다.
- **신호**: 신규 컴포넌트에 `useMemo`/`useCallback`이 등장한다. 빌드 시간이 눈에 띄게 늘어난다.

### R-005 · 디자인 토큰이 `globals.css` 한 파일에 집중되어 있다

- **상태**: 열림
- **영역**: 기술
- **확률**: 중 · **영향**: 하
- **내용**: Tailwind v4는 CSS-first 설정이라 `tailwind.config.*`가 없다. 토큰은 `src/app/globals.css`의 `@theme inline` 블록과 `:root`/`prefers-color-scheme: dark` 커스텀 프로퍼티에만 있다. 팀이 동시에 토큰을 추가하면 이 파일에서 충돌이 반복된다. 또 JS 설정을 전제한 서드파티 플러그인·예제가 그대로 동작하지 않는다.
- **대응**: 완화 — 토큰 추가 규칙(명명·다크 모드 쌍 유지)을 정하고, 라이트/다크 값을 항상 쌍으로 넣는다.
- **신호**: `globals.css`에서 머지 충돌이 반복된다. 다크 모드에서만 색이 빠진 컴포넌트가 나온다.

- **신호**: 고객 피드백에서 "모임"이 여전히 두 뜻으로 쓰인다. 화면 리뷰에서 "이 목록이 크루 목록인가 일정 목록인가"라는 질문이 반복된다.

### R-014 · Push가 이번 범위 밖인데 원문 №8이 push를 명시해 고객 기대와 어긋난다

- **상태**: 해소됨 (2026-07-23, D-004로 확정)

- **확률**: 상 · **영향**: 중
- **내용**: 원문 №8은 투표 종료 알림을 "(push, toast)"로 적었으나 원문 №13은 "push도 추후 개발 예정"이라 한다. **같은 문서 안에서 충돌한다.** 요구사항 문서는 №13을 우선해 toast만 in-scope로 두고 push를 FR-073(W)으로 미뤘다. 고객이 №8을 근거로 push를 기대하면 릴리스 시점에 범위 분쟁이 된다.

- **신호**: 고객이 "알림이 안 온다"고 말할 때 그 대상이 미접속 상태다. 릴리스 데모에서 push 시연을 요구받는다.

### D-006 · 캘린더 색은 Crew 단위로 결정론적 배정한다

- **일자**: 2026-07-23
- **결정자**: 고객 (요구사항 검토)

- **결정**: `paletteIndex = hash(crew.id) mod 12`로 **Crew 단위 결정론적 배정**한다. 팔레트 12색은 라이트·다크 각각 대비 3:1 이상을 사전 측정해 통과한 값만 등록하고, 오너는 팔레트 내에서 색을 직접 지정할 수 있다(자동 배정이 기본값). 같은 날짜 셀 안에서 색이 겹치면 나중 항목에 인접하지 않은 다음 인덱스를 임시 배정하며, **전역 유일성은 보장하지 않는다**(색 공간이 유한하므로 불가능하다). 색은 보조 신호일 뿐이고 바에는 크루명 텍스트와 `aria-label`을 반드시 병기한다.
- **대안**: ① **Meetup 단위 배정** — 원문 문구에는 가까우나 색의 정보 가치가 사라진다. ② **자동 배정 없이 개설 시 오너가 필수 선택** — 개설 플로우가 길어지고 미선택 상태 처리가 결국 필요하다.
- **영향**: 요구사항 3.5절, FR-062, `CrewLegend` 컴포넌트, NFR-018·NFR-019. `globals.css`의 `@theme inline`에 라이트/다크 색 쌍 12조 정의 필요(R-005). R-013 완화 근거. I-006 종결.

  - **`public`**: **비로그인 방문자를 포함해 누구나** 검색 결과에 노출되고 크루 소개(이름·설명·멤버 수·공개 범위)를 열람할 수 있다. 게시판·채팅·멤버 목록·캘린더는 크루원에게만 보인다.
  - **`private`**: **같은 크루의 크루원만** 검색·열람할 수 있다. 비소속 회원의 검색 결과에 나타나지 않으며, URL을 직접 알아도 크루명과 "초대 전용" 안내까지만 보인다. 가입은 초대로만 가능하다.
- **대안**: ① **3단계(공개/신청제/비공개)** — "검색은 되게 하되 내부는 감춘다"를 표현할 수 있으나, `public`이 이미 소개까지만 공개하므로 신청제와 실질 차이가 작다. ② **비로그인 열람 불가** — RLS가 "인증됨"을 기본 전제로 깔끔해지지만 신규 유입 경로가 사라진다.
- **영향**: FR-012·FR-014, 크루 관련 RLS 정책 전부. **비회원 접근 가능 화면이 생기므로** SC-07(크루 검색)·크루 소개 페이지에 비로그인 상태와 "가입하고 참여하기" 유도가 추가로 필요하다. 요구사항 3.3절 권한 매트릭스의 비회원 열 갱신 필요. I-008 종결.

### D-013 · Meetup은 날짜(필수)에 더해 시각·장소·정원을 가지며, 정원이 있으면 선착순 참석 신청을 받는다
