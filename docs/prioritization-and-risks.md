# 우선순위와 리스크

요구사항의 **우선순위**, **리스크 등록부**, **확정된 결정**을 기록한다.

- 이 파일의 6.3절 **결정 기록(D-\*)**이 확정 결정의 **단일 소스**다. **미결 이슈·개선사항은 여기가 아니라 [`ISSUES.md`](./ISSUES.md)**에 쓴다. 결정과 미결을 같은 곳에 두지 않는다.
- **다음 결정 번호: D-040** / **다음 리스크 번호: R-021** (등재할 때마다 이 줄을 갱신한다)
- 리스크는 **누구나 제보**한다. 미결이 확정되면 `ISSUES.md`의 I-\* 를 닫고 여기 6.3절에 D-\* 로 옮긴다.
- 항목은 상태가 바뀌어도 **지우지 않는다**. 같은 판단이 다시 올라왔을 때 이전 근거를 찾을 수 있어야 한다.

> **절 번호(6.x)는 요구사항 문서 묶음에서의 위치를 따른다.** 앞선 1~5절은 [`requirements/requirements.md`](./requirements/requirements.md)가 담당한다(2026-07-23 작성). 6.1절은 그 문서의 FR-\*/NFR-\* 를 근거로 채워졌다(R-008 참고).

## 6.1 우선순위

### 기록 형식

각 요구사항에 **MoSCoW 등급**과 **릴리스 목표**를 붙인다.

| 등급 | 뜻 | 판정 기준 |
| --- | --- | --- |
| **M** (Must) | 없으면 릴리스 불가 | 빠지면 제품이 목적을 달성하지 못한다 |
| **S** (Should) | 중요하나 우회 가능 | 수동 절차나 임시 대안으로 버틸 수 있다 |
| **C** (Could) | 여유가 있으면 | 빠져도 사용자가 목적을 달성한다 |
| **W** (Won't) | 이번 범위 밖 | 다음 릴리스로 명시적으로 미룬 것 |

```md
| ID | 요구사항 | 등급 | 릴리스 | 근거 | 의존 |
| --- | --- | --- | --- | --- | --- |
| FR-001 | 한 줄 요약 | M | v0.1 | 왜 이 등급인가 | FR-000, D-001 |
```

- **근거 없는 등급은 등재하지 않는다.** "중요해 보여서"는 근거가 아니다.
- 등급을 바꿀 때는 6.3절에 D-\* 결정을 남기고 여기서 그 번호를 참조한다.
- `W` 항목도 지우지 않는다. 무엇을 **의도적으로** 안 하기로 했는지가 범위 관리의 절반이다.

### R-001 · Next.js 16의 breaking change가 학습된 관습과 충돌한다

- **상태**: 열림
- **영역**: 기술
- **확률**: 상 · **영향**: 중
- **내용**: `next@16.2.11`은 이전 메이저와 API·규약이 다르다. 특히 요청 API 비동기화(`params`·`searchParams`·`cookies()`·`headers()`·`draftMode()`), `middleware.ts` → `proxy.ts` 대체, Turbopack 기본화 및 설정 위치(`next.config.ts` 최상위 `turbopack`), 캐싱 API 변경(`revalidateTag` 동작·`updateTag`·`refresh`·`cacheLife`/`cacheTag`), `next/image` 기본값 변경, 병렬 라우트 `default.js` 필수화. 관습대로 작성한 코드가 **런타임에서야** 깨질 수 있다.
- **대응**: 완화 — 코드 작성 전 `node_modules/next/dist/docs/`의 버전 대응 문서를 읽는다(`AGENTS.md` 규약). 업그레이드 가이드는 `node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md`.
- **신호**: 리뷰에서 동기 `params` 접근, `middleware.ts` 신규 생성, `experimental.turbo` 설정이 발견됨.

### R-002 · 테스트 러너·포매터·CI가 없어 회귀를 잡을 수단이 없다

- **상태**: 열림
- **영역**: 기술
- **확률**: 상 · **영향**: 상
- **내용**: `package.json`의 스크립트는 `dev`·`build`·`start`·`lint` 뿐이다. 자동 검증 수단은 사실상 **빌드 중 타입 검사**와 ESLint 뿐이고(`npx tsc --noEmit`로 단독 실행 가능), 동작 회귀를 잡는 층이 전혀 없다. 컴포넌트가 늘어날수록 수동 확인 비용이 선형으로 증가한다.
- **대응**: 완화 — 컴포넌트가 늘기 전에 테스트 러너 도입 여부를 D-\* 로 결정한다. 도입 전까지는 `/sample` 쇼케이스가 유일한 회귀 확인 지점이므로 신규 컴포넌트 등록을 강제한다.
- **신호**: 같은 컴포넌트에서 이미 고쳤던 버그가 재발한다. 리뷰가 "로컬에서 확인했다"에 의존한다.

### R-003 · Supabase가 설정만 있고 연결되지 않아 Mock↔실데이터 전환 비용이 미지수다

- **상태**: 열림 (2026-07-23 내용 정정 — 아래 참고)
- **영역**: 데이터
- **확률**: 중 · **영향**: 상
- **내용**: **[2026-07-23 정정]** 이 항목은 원래 "ref만 있고 스키마·마이그레이션이 없다"고 기술했으나 **사실이 아니었다.** 해당 ref는 **타 앱(축구 매니저 시뮬레이션)이 테이블 43개·마이그레이션 33건으로 점유** 중이었고, mo_im의 스키마는 애초에 존재한 적이 없다(D-018로 전용 프로젝트 신설 확정). 여기서 다루는 리스크는 **전환 경계의 부재**다 — mo_im에는 클라이언트 코드·데이터 접근 레이어가 없다. 개발 원칙은 "UI 컴포넌트 수정 없이 데이터 조회 부분만 교체"를 요구하지만, 그 경계(데이터 접근 레이어)가 아직 존재하지 않는다. 경계 없이 컴포넌트가 늘면 전환 시점에 **UI를 광범위하게 고쳐야 한다.**
- **대응**: 완화 — Mock과 실데이터가 **동일한 TypeScript 타입**을 쓰도록 타입을 먼저 정의하고, 조회 함수를 컴포넌트 밖으로 분리한다(**D-030**이 경계 4개를 규약화했다). 스키마 확정 후에는 `mcp__supabase__generate_typescript_types`로 타입을 생성해 수기 타입과 대조한다.
- **신호**: 컴포넌트 안에서 Mock 배열을 직접 import하거나, Mock 전용 필드가 타입에 섞인다.

### R-004 · React Compiler와 수동 메모이제이션이 충돌한다

- **상태**: 열림
- **영역**: 기술
- **확률**: 중 · **영향**: 중
- **내용**: `next.config.ts`에 `reactCompiler: true`가 켜져 있고 `babel-plugin-react-compiler@1.0.0`을 쓴다. 습관적으로 넣은 `useMemo`/`useCallback`/`memo`는 불필요하며 컴파일러 최적화와 충돌할 수 있다. 또한 Babel 경유 변환이라 빌드 시간에 영향을 준다.
- **대응**: 수용 + 완화 — 수동 메모이제이션을 금지 관습으로 두고 리뷰에서 걸러낸다. 빌드 시간이 문제가 되면 그때 D-\* 로 재검토한다.
- **신호**: 신규 컴포넌트에 `useMemo`/`useCallback`이 등장한다. 빌드 시간이 눈에 띄게 늘어난다.

### R-005 · 디자인 토큰이 `globals.css` 한 파일에 집중되어 있다

- **대응**: 완화 — 토큰 추가 규칙(명명·다크 모드 쌍 유지)을 정하고, 라이트/다크 값을 항상 쌍으로 넣는다.
- **신호**: `globals.css`에서 머지 충돌이 반복된다. 다크 모드에서만 색이 빠진 컴포넌트가 나온다.

### D-006 · 캘린더 색은 Crew 단위로 결정론적 배정한다

### D-019 · 정원 원자성은 카운터 + 조건부 UPDATE로 보장하며 RLS로 시도하지 않는다

- **결정**: `meetup` 테이블에 **`attending_count` 카운터**를 두고, 참석 신청을 `update meetup set attending_count = attending_count + 1 where id = ? and attending_count < capacity` **조건부 UPDATE**로 처리한다(행 락 획득 후 최신 행으로 WHERE를 재평가하므로 원자적). 보강으로 `check (attending_count <= capacity)` 와 **`unique(meetup_id, profile_id)`** 를 건다. 유일 제약은 FR-067 E2의 멱등성(upsert) 전제이기도 하다. **RLS 정책으로 정원을 강제하지 않는다.**
- **대안**: 비노출 스키마의 `SECURITY DEFINER` RPC + `select … for update` — 동등하게 유효하며 D-028·네이티브 전환 방향과도 맞는다. 조건부 UPDATE 쪽이 왕복이 적어 기본안으로 택했고, 참석 로직이 복잡해지면 RPC로 옮긴다.
- **영향**: FR-066 AC2·E2, FR-067 E2, D-013. 요구사항 5.2절에 `attending_count`·유일 제약 추가. **부하 상황에서만 조용히 실패하므로 R-002(수동 QA)로는 검출되지 않는다** — 동시성 테스트가 필요하다.

### D-023 · Realtime은 Broadcast를 쓴다 (Postgres Changes 아님)

- **대안**: **Postgres Changes** — 설정이 간단하고 트리거를 쓰지 않아도 되나, NFR-003(채팅 전달 p95 ≤1초)을 크루당 100세션(NFR-006)에서 만족한다는 근거가 없다.

  - **① 채도는 데이터만 쓴다** — 유채색은 크루 12색이 전부 가져가고 UI 크롬은 잉크 뉴트럴(H=258, C≤0.022)로 물러난다. 활성 내비 표시를 배경 칠에서 잉크 바로 바꾼 것도 색 면적을 크루 식별에 남기기 위해서다. 예외는 `--destructive`와 크루 팔레트뿐이다.
  - **② 배경·카드 표면의 밝기는 재량이 아니다** — 12색 대비 검증(D-026)이 라이트 `oklch(1 0 0)` / 다크 `oklch(0.145 0 0)` 기준이고 최저치가 3.26이라 여유가 없다. 색조는 배경이 아니라 표면·경계·텍스트에서 낸다. **실측 결과 `--muted`(`oklch(0.97)`) 위에서는 crew-3·crew-6이 2.99로 3:1을 깬다 — 이건 개편이 만든 제약이 아니라 원래 있던 함정이며, "크루색은 `--background`·`--card` 위에만 놓는다"를 규칙으로 박았다.**
  - **시그니처는 확정성 스케일**이다(`certainty-draft`/`pending`/`confirmed`). 제품 한 줄 정의("채팅에 묻힌 약속을 투표로 확정해 캘린더에 남긴다")를 채도에 실어 **확실할수록 색이 차게** 한다. 색 단독이 아니라 점선/실선/채움이라는 형태 차이가 함께 간다(WCAG 1.4.1).
  - **채움 위 글자색은 색마다 다르다** — 12색 휘도가 0.1365~0.2725라 한 가지 글자색으로는 4.5:1을 못 맞춘다. 어느 쪽을 쓸지는 `crew-palette.ts`의 **`textOn` 필드가 데이터로** 들고 있고 `crewCertaintyVars()`가 채움색과 짝으로만 돌려준다(짝을 깨는 실수는 라이트 모드 테스트로 드러나지 않는다).
  - **타이포는 Noto Sans KR(가변 wght 100~900) + Geist Mono**다. 모노는 코드용이 아니라 수·시각의 서체로 격상하되 **한글이 섞인 문자열에는 쓰지 않는다**(한글 글리프가 없어 한 줄에서 서체가 갈린다). `word-break: keep-all`로 한글을 낱말 단위로 끊는다.
    - **가변을 쓰는 것은 실측 근거가 있다** — 고정 400+600은 조각 248개·6,874KB인데 가변 100~900은 **조각 124개·3,437KB**로 절반이면서 굵기 전 범위를 준다. 고정 굵기 하나만 써도 가변과 같은 비용이라 이 폰트에서 가변은 언제나 이득이다.
    - **[2026-07-24 후속 — 고객 지시]** 최초 개편은 **IBM Plex Sans KR(고정 400·600)** 로 갔다가 **Noto Sans KR 가변으로 교체**했다. IBM Plex 쪽은 라틴 IBM Plex와 한 가족이라 워드마크와의 조화가 좋았으나 **가변 판이 없어** 굵기마다 글리프 한 벌이 필요했다. 교체 후 한 페이지 기준 실제 로드 조각이 **30개 → 20개**로 줄었다(굵기별 중복 로드 소멸).
  - **`preload: false`** — 한글 폰트는 유니코드 조각 수십~수백 개로 배포돼 preload를 켜면 초기 문서에 폰트 요청 수십 건이 박힌다(IBM Plex 시절 실측 47건). 끄면 브라우저가 `unicode-range`로 실제 쓰인 조각만(현재 실측 20개) 받는다.
- **대안**: **㉠ 크림 배경 + 세리프 디스플레이 + 테라코타 액센트** — 배경 밝기를 바꾸는 순간 12색 대비 검증 전체가 무효가 된다(휘도 창이 두 배경에 딱 맞춰져 있다). 애초에 선택지가 아니었다. **㉡ 하드라인·명조 디스플레이의 "기록·원장" 방향** — "채팅에 묻힌 것을 기록으로 남긴다"는 서사와 적합도는 가장 높으나, 360px 터치 UI(NFR-026·027)에서 거칠어진다. **㉢ 토큰·폰트만 교체** — 셸의 손수 구현과 접근성 결함(위 ③④)이 그대로 남는다.
- **영향**: `src/app/globals.css`(토큰·`@utility certainty-*`·조판 기본값) · `src/app/layout.tsx`(폰트) · `src/lib/crew-palette.ts`(`textOn`·`contrastOnFill`·`crewCertaintyVars`) · `src/lib/strings/ko.ts`(`a11y.unreadCount`) · 셸 4종 · `/sample`(Task 012 구조로 확장) · `src/components/ui/` 7종 추가(badge·skeleton·tabs·separator·toggle/toggle-group·empty·avatar). **D-026은 그대로 유효하며 이 결정이 그 위에 얹힌다.** 전체 근거·수치는 [`design/design-language.md`](./design/design-language.md). **I-020(ThemeProvider 부재)은 해소되지 않았다** — 다크 블록 두 벌을 손으로 동기화하는 상태가 유지된다.

### D-039 · 크루원 게이트는 `(app)/crews/[crewId]/layout.tsx`가 라우트 레벨에서 한 번 막고, 컨테이너의 세밀한 판정은 그대로 둔다

- **일자**: 2026-07-24
- **결정자**: CREW (Task 016B, I-035 후속 처리)
- **맥락**: I-035는 `(app)/crews/[crewId]/*`(board·chat·members·settings)가 "로그인 여부"만 검사하고 "그 크루의 멤버인가"는 라우트 레벨에서 검사하지 않는다고 지적했다. 지금까지는 `BoardListContainer`의 `resolveBoardViewer`+`checkPermission("board:read")`만 컨테이너 레벨에서 비멤버를 막았다 — 새 하위 화면이 추가될 때마다 이 방어를 각자 다시 구현해야 하고, 하나라도 빠뜨리면 그 라우트만 조용히 뚫린다는 것이 위험이었다. I-035 「후속」이 제시한 유력안(동적 세그먼트 `layout.tsx`에서 멤버십을 한 번 조회해 하위 전체에 적용)을 검토해 채택했다.
- **결정**:
  1. **`src/app/(app)/crews/[crewId]/layout.tsx`를 새로 만든다.** `getAuthSession()`+`assertAuthenticatedSession`으로 세션을 좁힌 뒤(`(app)` 레이아웃이 이미 인증을 보장하지만 타입은 다시 좁혀야 한다, D-030 ④의 "허용되는 재작성" 패턴과 동일) `getCrewById`로 크루 존재를, `getCrewMembership`+`isActiveMembership`(`lib/rules/crew-membership-transition.ts`)로 **활성 크루원인지만** 판정한다. 크루가 없으면 `notFound()`, 활성 크루원이 아니면 `cause: { code: "forbidden" }`를 실어 던진다 — 전역 `src/app/error.tsx`의 `classifyError`가 이를 `RouteErrorBoundary(kind="forbidden")`로 그린다(`board:read` 거부와 동일한 표현).
  2. **판정은 "활성 크루원인가" 하나뿐이다** — `board:read`처럼 액션별 세밀한 매트릭스 조건(예: 관리자의 게시판 열람 허용)은 이 레이아웃이 흉내 내지 않는다. 지금 `AuthSession`에는 `system_admin`을 표현하는 필드가 아예 없어(Task 042A 이전) 이 근사가 안전하다 — 관리자 세션이 실제로 생기면 이 레이아웃의 조건을 다시 검토해야 한다(그때까지는 아래 3번이 그 세밀한 차이를 계속 책임진다).
  3. **`resolveBoardViewer` + `BoardListContainer`의 `board:read` 컨테이너 방어는 제거하지 않고 그대로 둔다.** 두 가지 이유다. 첫째, 그 컨테이너는 "크루원인가"만이 아니라 `role`(member/staff/owner 세분)을 계산해 `post:create`(글쓰기 버튼 노출) 판정에도 쓴다 — 이 레이어의 목적 자체가 게이트 하나가 아니다. 둘째, 새 세밀한 판정(예: 향후 관리자 열람 허용)이 필요해지면 액션별 컨테이너가 여전히 그 자리다 — 레이아웃은 "누가 여기 도달할 수 있는가"라는 조 굵은 경계만 지키고, "그 사람이 정확히 뭘 할 수 있는가"는 원래도 컨테이너·Server Action 몫이었다(D-030 ④ "`(app)`은 인증 게이트이지 크루원 게이트가 아니다"와 대칭 — 이 레이아웃도 "board:read 게이트"가 아니라 "크루원 게이트"일 뿐이다).
- **대안**: **컨테이너 방어만 유지, 레이아웃을 만들지 않는다(현상 유지)** — I-035가 지적한 "반복하면 하나를 빠뜨린다" 위험이 그대로 남는다. **레이아웃이 액션별 세밀한 판정까지 흉내 낸다** — 관리자 열람처럼 화면마다 다른 예외를 레이아웃 하나가 전부 알아야 해서 오히려 판정이 한곳에 뭉치는 대신 레이아웃 자체가 비대해진다 — 채택하지 않았다.
- **영향**: `docs/CONVENTIONS.md` D-030 ④ 절 갱신(아래 "새 보호 페이지를 추가할 때" 문단에 이 레이아웃 언급 추가). I-035 종결. `(app)/crews/new`(형제 세그먼트)와 `/crews`·`/crews/[crewId]`(`(app)` 밖, D-007)는 이 레이아웃의 대상이 아니다 — 각각 "크루원 개념이 아직 없음"·"게스트도 접근 허용"이라는 정반대 이유로 제외된다.
