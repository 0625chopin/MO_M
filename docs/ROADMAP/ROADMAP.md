# mo_im 개발 로드맵

채팅에 묻히는 모임 일정을 **게시글 기반 찬반 투표로 확정하고, 가결된 일정을 크루 색으로 구분된 통합 캘린더에 자동 등록**하는 크루 모임 웹 서비스를 만든다.

## 개요

mo_im은 동호회·소모임을 운영하거나 참여하는 일반 성인 사용자를 위한 서비스로, 다음을 제공한다.

- **크루와 멤버십**: 크루 개설(즉시 생성, D-008), 핸들 검색 기반 초대, 가입 신청·승인, 임원 임명
- **커뮤니티와 투표**: 모임 제안글 등록 시 찬반 투표 자동 생성, 정족수 1/3 + 찬성>반대로 가결 판정(D-003)
- **통합 캘린더**: 가결된 Meetup을 크루 색으로 구분해 월간 뷰에 자동 등록, 크루 필터·날짜 상세 패널·참석 신청
- **실시간 채팅**: 크루당 채팅방 1개, 게시글 링크 카드 공유와 즉시 이동
- **알림**: 인앱 토스트 + 알림 센터(Web Push는 이번 범위 밖, D-004)

### 이 문서의 위치와 규칙

- 요구사항의 단일 소스는 [`docs/requirements/requirements.md`](../requirements/requirements.md)(FR-\*/NFR-\*/CON-\*), 우선순위·리스크·**확정 결정(D-\*)**의 단일 소스는 [`docs/prioritization-and-risks.md`](../prioritization-and-risks.md), 미결 이슈(I-\*)의 단일 소스는 [`docs/ISSUES.md`](../ISSUES.md)다. **이 로드맵과 위 문서가 어긋나면 위 문서가 이긴다.**
- 착수용 재구성본은 [`docs/prd/PRD.md`](../prd/PRD.md), 기술 검증·공수 산정 근거는 [`docs/prd/PRD-validation.md`](../prd/PRD-validation.md)에 있다.
- **일정(날짜·인일·기간)은 이 문서에 적지 않는다.** 팀 규모가 미정이라 기간으로 환산할 수 없다(**I-019**). 인일 산정치는 `PRD-validation.md` §6에 있고, 인원이 정해지면 `schedule-planner`가 별도 일정을 산출한다.
- **진행률·완료 개수 같은 갱신되는 집계값은 적지 않는다.** 상태는 각 Task 줄의 ✅ 표시 하나로만 관리한다(R-006 재발 방지).

## 개발 워크플로우

1. **작업 계획**
   - 기존 코드베이스를 학습하고 현재 상태를 파악한다. **경로를 참조하기 전에 실제 파일 존재 여부를 먼저 확인한다**(R-006).
   - 새로운 작업을 포함하도록 이 `ROADMAP.md`를 갱신한다. 우선순위 작업은 마지막 완료 Task 다음에 삽입한다.
   - 구현 착수 전 해당 FR/NFR 원문과 관련 D-\*를 먼저 읽는다. **D-\*를 거스르는 판단이 필요하면 코드가 아니라 `prioritization-and-risks.md` 6.3절에 새 D-\*를 남긴다.**

2. **작업 생성**
   - `docs/ROADMAP/tasks/` 에 작업 파일을 만든다. 명명 형식은 `XXX-description.md`(예: `001-conventions.md`).
   - 고수준 명세, 관련 파일, 수락 기준, 구현 단계, **참조(FR/NFR/D/R/I)** 를 포함한다.
   - **API 연동·비즈니스 로직 작업 파일에는 `## 테스트 체크리스트` 섹션을 반드시 둔다**(Playwright MCP 시나리오).
   - 형식 예시는 직전 완료 Task 두 건을 참조한다(예: 012 작성 시 011·010). 완료 Task는 체크된 박스와 변경 요약을 담고 있으므로, **새 작업 파일은 빈 박스에 변경 요약 없이** 시작한다. 초기 상태 샘플은 `docs/ROADMAP/tasks/000-sample.md`로 만들어 둔다.

3. **작업 구현**
   - 작업 파일의 명세를 따른다.
   - Next.js 코드를 쓰기 전에 `node_modules/next/dist/docs/`의 버전 문서를 읽는다(`AGENTS.md`, CON-02, **R-001**).
   - 새 컴포넌트는 **만든 그 커밋에서** `/sample`에 4상태로 등록한다(CON-09). 등록을 미루면 쇼케이스가 곧 실물과 어긋난다.
   - **API 연동·비즈니스 로직은 Playwright MCP로 검증한다.** 각 단계 후 작업 파일의 진행 상황을 갱신하고, 구현 완료 후 E2E 시나리오를 실행해 통과를 확인한 뒤 다음 단계로 넘어간다.
   - 각 단계 완료 후 중단하고 추가 지시를 기다린다.

4. **로드맵 업데이트**
   - 완료된 Task를 ✅로 표시하고 `See: docs/ROADMAP/tasks/XXX-xxx.md` 참조를 붙인다. Phase의 모든 Task가 완료되면 Phase 제목에도 ✅를 붙인다.

## 개발 순서를 지배하는 제약

이 로드맵의 Phase 구성은 취향이 아니라 아래 확정 결정에서 나온다.

- **Mock First (`CLAUDE.md`)** — 더미 데이터로 화면을 먼저 완성하고 **그 다음에** DB를 설계·연결한다. 그래서 v0.1 전체가 Mock이고 Supabase는 Phase 4(v0.2)에서 처음 등장한다. 릴리스 정의도 이와 같다: **v0.1 = Mock 기반 화면·컴포넌트 완성 / v0.2 = Supabase 연결·실데이터 CRUD·인증·RLS / v1.0 = 정식 릴리스**.
- **D-030 전환 경계 4가지** — ①표현/컨테이너 분리 ②`subscribeToRoom(id, onEvent): Unsubscribe` 구독 추상화 ③`/sample` 4상태의 "오류"에 도메인 오류(RLS 403·정원 마감·동시 수정 충돌) 포함 ④인증 경계는 **레이아웃**에서 처리. 넷 다 v0.1부터 지킨다(Task 007·008·011·012). 전환 시점에 리팩터링하는 것은 컴포넌트 약 70종이 만들어진 **가장 비싼 시점**이다.
- **D-011** — v0.1은 한국어 단독이며 **경로에 로케일 세그먼트를 두지 않는다**(`/crews`, `/calendar`, `/sample`). `src/app/[lang]/`·`src/proxy.ts` 로케일 리다이렉트는 **범위 밖이므로 Task로 만들지 않는다**(NFR-024, 등급 W). 대신 문자열 분리(NFR-023, Task 003)와 게시글 링크의 리소스 ID 기준 저장(FR-052, Task 020)은 v0.1 Task다.
- **D-029 / React Compiler** — 수동 `useMemo`/`useCallback`/`memo`를 기본 금지하되, **측정 근거를 남기면 `"use no memo"` 예외를 허용**한다. INP 목표는 메모이제이션이 아니라 렌더링 전략(윈도잉·안정적인 `key`·`startTransition`·사전 인덱싱)으로 달성한다.
- **접근성·아키텍처 경계가 v0.1에 몰려 있다** — NFR-017~022(접근성)와 NFR-034~036(경계)은 **나중에 소급 적용하는 비용이 처음부터 지키는 비용보다 훨씬 크다.** 뒤로 미루지 않는다(R-020의 신호가 정확히 "접근성·4상태를 나중으로 미루기 시작한다"이다).
- **R-002** — 테스트 러너·CI가 없어 회귀를 잡을 자동 수단이 없다. 도입 전까지 `/sample` 쇼케이스가 유일한 회귀 확인 지점이므로 신규 컴포넌트 등록을 강제한다. 러너 도입 여부는 Task 024에서 D-\*로 결정한다.

- **Task 001: 디렉터리 구조·명명 규약·린트 규칙 확정** - 우선순위

## Phase 2: 애플리케이션 골격 (v0.1)

> **왜 이 순서인가**: 구조 → UI → 기능. 이 Phase는 **화면 하나도 완성하지 않는다.** 라우트·타입·데이터 경계·구독 추상화·앱 셸·`/sample` 셸을 먼저 세워, Phase 3의 화면 Task들이 서로 독립적으로 병렬 진행되게 만드는 것이 목적이다. `PRD-validation.md` §6이 관측한 대로 **v0.1 비용의 약 29%가 어떤 FR에도 속하지 않는 공통 기반**이며, 이 구간은 화면 진척이 눈에 보이지 않는다(**R-020**). 건너뛰자는 제안이 나오면 그것이 R-020의 신호다.

## 로드맵이 주시하는 리스크 신호

| 리스크 | 신호 — 이게 보이면 대응이 새고 있는 것이다 |
| --- | --- |
| **R-001** Next.js 16 breaking change | 리뷰에서 동기 `params` 접근, `middleware.ts` 신규 생성, `experimental.turbo` 설정이 발견된다 |
| **R-002** 검증 수단 부재 | 이미 고쳤던 버그가 재발한다. 리뷰가 "로컬에서 확인했다"에 의존한다 |
| **R-003 / R-015** 경계 누수 | 컴포넌트가 Mock 배열이나 Supabase 클라이언트를 직접 import한다. 투표 판정이 서버 컴포넌트 본문에 인라인된다. 순수 로직 모듈이 React를 import한다 |
| **R-004** React Compiler 충돌 | 신규 컴포넌트에 근거 없는 `useMemo`/`useCallback`이 등장한다. 빌드 시간이 눈에 띄게 는다 |
| **R-005** 토큰 집중 | `globals.css`에서 머지 충돌이 반복된다. 다크 모드에서만 색이 빠진 컴포넌트가 나온다 |
| **R-006** 문서와 실물 괴리 | 없는 파일을 수정하려는 시도. 새 컴포넌트가 `/sample`에 등록되지 않은 채 커밋된다. 문서에 진행률·개수·완료형 서술이 다시 등장한다 |
| **R-012** 핸들 검색 노출 | 검색 API가 3필드 외를 반환한다. 부분·접두사 검색이 추가된다. 응답 시간이 계정 존재 여부에 따라 달라진다 |
| **R-013** 팔레트 대비 | 대비 측정 없이 색을 추가한다. 다크 모드에서만 바가 배경에 묻힌다. 캘린더 바에 텍스트 라벨이 없다 |
| **R-016** 로케일 경로 부재의 잔여 비용 | 컴포넌트에 한국어 문자열이 직접 박힌다. 채팅 링크가 절대 URL 문자열로 저장된다 |
| **R-017** 크루 수 상한 없음 | 캘린더에 크루 필터가 없다. 색 배정이 12색 초과를 처리하지 않는다. QA가 크루 20개 계정을 만들어 보지 않는다 |
| **R-020** 공수 초과 압력 | 공통 기반 작업 중 화면 진척이 없다고 건너뛰자는 제안이 나온다. **접근성·4상태를 "나중에"로 미루기 시작한다** |
